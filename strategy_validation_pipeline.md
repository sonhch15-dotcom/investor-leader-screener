# 전략 검증 자동화 절차

## 목적

새 전략을 만들 때마다 기준과 결과가 바뀌는 문제를 막고, 같은 순서로 빠르게 검증하고 대시보드에 남기기 위한 절차입니다.

## 1. 연구 계약을 먼저 고정

```powershell
npm run study:new -- my-strategy "전략 이름" US_STOCK
```

생성된 `studies/my-strategy/study.json`에서 다음 항목을 백테스트 전에 확정합니다.

- 당시 구성 종목을 쓰는지, 현재 생존 종목만 쓰는지
- 섹터와 업종 분류 체계 및 버전
- 신호일, 실제 매수일, 데이터 지연일
- 초기 자금, 환율 기준일과 출처, 수수료, 환전 비용
- 종목별 한도, 소수점 거래 여부, 매도 규칙
- 기존 전략과 후보 전략의 정확한 키

이 계약을 바꾸면 같은 백테스트의 재실행이 아니라 새로운 실험으로 기록합니다.

## 2. 공통 검증 묶음

모든 후보는 최소한 아래를 실행합니다.

1. 전체 기간과 서로 다른 시작 시점
2. 최근 구간과 약세장 구간
3. 거래비용 0.10%와 0.25%, 환전 비용 스트레스
4. 신호 후 체결 지연과 구성 종목 데이터 지연
5. 상장폐지 종목 포함 여부
6. 가장 큰 수익 종목 1~2개를 제외한 결과
7. 초기 1억원 한정 계좌의 매수 참여율과 평균 현금 비중
8. 기준 지수와 기존 공식 전략을 같은 자금 조건으로 비교

## 3. 승격 기준

- 매수 참여율 95% 이상
- 독립 시작 구간에서 기존 전략보다 반복적으로 우수
- 최대 손실폭이 기존보다 3%p 넘게 악화되지 않음
- 한두 종목을 제거해도 결론이 완전히 뒤집히지 않음
- 최소 6개월, 권장 12개월의 미래 관찰 결과 확보

누적 수익률 한 항목만 좋아서는 공식 전략으로 승격하지 않습니다.

## 4. 자동 검사와 보고서 등록

```powershell
npm run study:verify
npm run build:reports
npm run verify:reports
npm run build:pages
```

최종 보고서를 `config/report-catalog.json`에 한 줄 등록하면 전략 규칙 탭의 연구 보고서 보관함과 GitHub Pages 빌드에 함께 연결됩니다.

## 5. 앱 반영 순서

1. public 저장소에서 검증과 승격 결정을 완료합니다.
2. 보고서와 `THREAD_SHARED_CONTEXT.md`에 결정 근거를 기록합니다.
3. signal package에서 단일 active 전략과 전환 월을 명시합니다.
4. Android 앱은 종목을 다시 계산하지 않고 public API 신호만 실행합니다.
5. 기존 lot은 매수 당시 전략 키와 매도 일정을 유지합니다.

## 재현성 원칙

- 이전 결과를 덮어쓰지 않고 연구 ID와 데이터 기준일을 남깁니다.
- 종목 분류와 유니버스가 달라지면 기존 결과와 직접 연결하지 않습니다.
- 결과가 불리해도 삭제하지 않고 `research`, `candidate`, `paused`, `active`, `retired` 상태로 관리합니다.
