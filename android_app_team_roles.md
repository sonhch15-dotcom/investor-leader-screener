# Android App Team Roles

작성일: 2026-07-09

이 문서는 Investor Run Android 앱을 업데이트하거나 새 기능을 설계할 때 반드시 따를 팀 역할과 검토 규칙을 정의한다.  
이 프로젝트에서 "팀"은 실제 인원이라기보다, Codex가 개발할 때 반드시 분리해서 수행해야 하는 책임 관점이다.

목표는 사용자가 매번 문제를 하나씩 짚어줘야 하는 상황을 줄이고, 앱이 실전 투자 운용 도구로 맞는지 먼저 검토한 뒤 구현하는 것이다.

Agent coding constitution: `android_app_agent_coding_rules.md`

## 1. Product / 사용자 운용 팀

역할:

- 앱이 "전략 검증 결과를 보여주는 웹"이 아니라 "전략을 실전 운용하도록 돕는 Android 앱"인지 계속 확인한다.
- 사용자의 실제 운용 구조를 기준으로 기능을 판단한다.
- 미국 주식, 한국 주식, 연금 ETF 계좌가 서로 분리 운용되면서 총자산에서는 통합되는 구조를 유지한다.

반드시 확인할 것:

- 새 기능이 사용자의 실제 매수/매도/기록/검증 흐름에 도움이 되는가.
- 계좌별 운용 자금, 통화, 전략이 섞이지 않는가.
- 총자산 화면에서는 계좌별 자산이 원화 기준으로 자연스럽게 통합되는가.
- 기능이 "보기 좋은 화면"에 그치지 않고 실제 주문 행동으로 이어지는가.

실패 사례에서 배운 점:

- 하나의 예수금 입력 시스템은 실제 3계좌 운용 구조에 맞지 않았다.
- 웹에서 보던 검증 결과를 그대로 앱에 옮기는 것은 실전 앱 목적과 어긋난다.

## 2. UX 팀

역할:

- 사용자가 앱을 열고, 전략 신호를 확인하고, 주문하고, 체결을 기록하고, 검증받는 전체 흐름을 설계한다.
- 입력 피로를 줄이고, 사용자가 실수했을 때 빠르게 수정할 수 있는 구조를 만든다.

반드시 확인할 것:

- 텍스트 입력을 최소화했는가.
- 금액 입력은 `1,000,000`처럼 자동 콤마가 적용되는가.
- 계좌, 운용 섹션, 전략 선택은 미니탭/선택 컨트롤로 빠르게 접근 가능한가.
- 주문 가이드에서 체결 기록까지 흐름이 끊기지 않는가.
- 6개월/12개월/주봉 훼손 같은 lot 이벤트가 매도 체결 기록으로 바로 이어지는가.
- 사용자가 목표 비중을 정확히 맞추지 못하는 현실을 허용 기준으로 반영했는가.
- 하단 Android 시스템 버튼, 노치, 상태바, 하단 내비게이션과 화면 요소가 겹치지 않는가.

실패 사례에서 배운 점:

- 입금/출금과 체결 기록이 텍스트 입력 중심이면 앱 사용성이 떨어진다.
- 운용 화면에서 미국/한국/ETF가 세로로 모두 나열되면 복잡하다. 미니탭이 더 적절하다.
- "전략 잠시 중지"처럼 목적이 불명확한 버튼은 제거해야 한다.
- 버튼이 과하게 크면 정보 밀도와 전문성이 떨어진다.

## 3. Design / 시각 시스템 팀

역할:

- 개인용 앱이어도 출시 제품에 가까운 완성도를 목표로 시각 품질을 관리한다.
- 자산, 전략, 위험, 알림 상태가 빠르게 읽히는 화면을 만든다.

반드시 확인할 것:

- 카드, 버튼, 미니탭, 차트의 시각 위계가 명확한가.
- 폰 화면에서 텍스트가 잘리거나 겹치지 않는가.
- 도넛/파이 차트가 카드 안에서 잘리지 않는가.
- 금액, 수익률, 비중, 경고가 한눈에 구분되는가.
- 과한 장식보다 자산관리 앱다운 차분함과 가독성을 우선하는가.

실패 사례에서 배운 점:

- 도넛 차트의 상하단이 잘리는 문제는 디자인 QA에서 먼저 잡았어야 한다.
- 카드 안 정보가 많아질 때 버튼과 차트 크기를 조정하지 않으면 화면이 엉성해 보인다.

## 4. 자산 운용 / 전략 검증 팀

역할:

- 앱에 들어간 주문금액, 비중, 매도/리밸런싱 계산이 실제 검증된 전략과 맞는지 책임진다.
- "추천 종목 표시"와 "실제 얼마를 사야 하는가"는 별개의 문제로 보고 검증한다.

반드시 확인할 것:

- 미국 `Leader2 + Repeat Theme Combo Cap27.5`는 단순 2분할이 아니라 다음을 반영하는가.
  - 계좌 평가 원금 기준 기본 매수 비율
  - 반복 추천 이력
  - AI/하드웨어 테마 가중
  - 종목당 누적 원금 상한 27.5%
- 한국 주식 `Leader2`는 다음을 반영하는가.
  - 초기 3개월 월 30%
  - 이후 월 15%
  - 추천 종목 수 기준 분할
  - 종목당 누적 원금 상한 22.5%
  - 1주 단위 주문 현실
- ETF `Core/Satellite/Defense 50/40/10`은 다음을 반영하는가.
  - 계좌 평가금액 기준 목표금액
  - 현재 보유 평가액
  - 매수/매도 차이
  - 실제 주문 가능 수량
  - 허용 오차
- 주식 계좌 lot 매도 정책은 다음을 반영하는가.
  - 매수 기록별 lot 생성
  - 매도 기록의 FIFO 차감 또는 사용자가 선택한 lot 차감
  - 6개월 50% 매도 검토
  - 12개월 전량 매도 검토
  - 주봉 훼손 시 잔여 lot 매도 검토
- ETF 리밸런싱과 주식 lot 만기 매도 규칙이 서로 섞이지 않는가.

실패 사례에서 배운 점:

- 미국 계좌 8만 달러에서 추천 2종목에 반반 투입하라는 가이드는 `Cap27.5` 전략과 맞지 않았다.
- 한국 주식 추천은 들어가 있었지만 주문금액이 단순 2분할이면 실전 운용 앱으로 부족하다.
- ETF 목표비중만 보여주고 실제 매수/매도 수량을 계산하지 않으면 실전 리밸런싱 도구가 아니다.

## 5. 데이터 / 자동화 팀

역할:

- 신호, 가격, 환율, 주봉 이탈 데이터가 앱에 안정적으로 들어오도록 책임진다.
- APK 내장 데이터와 원격 데이터의 우선순위, 캐시, 업데이트 정책을 관리한다.

반드시 확인할 것:

- `signals/latest.json`, `prices/latest.json`, `fx/latest.json`, `weekly-trends/latest.json`가 APK에 포함되어 있는가.
- 원격 데이터가 있더라도 APK 내장 데이터보다 오래된 패키지가 앱 상태를 덮어쓰지 않는가.
- 환율과 현재가가 자동 업데이트 가능한 구조인가.
- 데이터 지연이나 실패가 있을 때 앱이 중단되지 않고 사용자에게 상태를 알려주는가.
- 신호 패키지 schema가 바뀌면 앱 파서와 문서가 함께 업데이트되는가.

실패 사례에서 배운 점:

- 기존 앱 위에 새 APK를 설치해도 앱 내부 캐시가 남아 있으면 예전 추천이 계속 보일 수 있다.
- 앱 시작 후 자동 원격 동기화가 오래된 서버 데이터를 다시 저장하면 업데이트가 된 것처럼 보이지 않는다.

## 6. Android 개발 팀

역할:

- 위 팀들의 요구를 실제 Android 코드로 구현한다.
- 당장 개인용 APK라도 업데이트, 저장, 화면 안전영역, 알림, 입력 흐름은 제품 수준으로 다룬다.

반드시 확인할 것:

- `versionCode`, `versionName`, APK 파일명이 일치하는가.
- 기존 앱 위에 업데이트 설치했을 때 데이터가 꼬이지 않는가.
- 하위 호환이 필요한 기존 저장 데이터가 유지되는가.
- 새 계산 로직이 기존 보유/현금/체결 기록 구조와 충돌하지 않는가.
- 디버그 APK라도 설치 가능한 산출물까지 만들어졌는가.

실패 사례에서 배운 점:

- APK만 새로 만들어도 내부 캐시 정책을 고려하지 않으면 사용자는 예전 앱처럼 느낀다.
- 기능 구현 후 실제 업데이트 설치 시나리오를 검증해야 한다.

## 7. QA / 검증 팀

역할:

- 사용자가 발견하기 전에 화면, 계산, 데이터, 업데이트 문제를 먼저 찾는다.
- 단순 빌드 성공이 아니라 실제 운용 시나리오 기준으로 검증한다.

반드시 확인할 것:

- 빌드가 성공했는가.
- APK 내부에 필요한 asset이 포함되어 있는가.
- 대표 계산 케이스를 손으로 또는 스크립트로 재검산했는가.
- 미국 8만 달러, 한국 1천만원, ETF 1천만원 같은 기준 케이스가 전략과 맞는가.
- 기존 앱 위에 업데이트 설치했을 때 새 데이터가 보이는가.
- 화면에서 차트/버튼/내비게이션이 겹치지 않는가.

필수 검증 예시:

- 미국 80,000 USD 계좌:
  - TECH 약 8,000 USD
  - STX 약 10,000 USD
  - 종목당 상한 22,000 USD
- 한국 주식 10,000,000 KRW 계좌:
  - 초기 구간 추천 2종목이면 각 1,500,000 KRW 목표
  - 종목당 상한 2,250,000 KRW
- ETF 10,000,000 KRW 계좌:
  - 50% 목표 5,000,000 KRW
  - 40% 목표 4,000,000 KRW
  - 10% 목표 1,000,000 KRW

## 8. 문서 / 릴리즈 팀

역할:

- 코드와 APK만 만들지 않고, 왜 바꿨는지와 어떤 버전에서 바뀌었는지 기록한다.
- 다음 개발 때 같은 실수를 반복하지 않도록 문서에 남긴다.

반드시 업데이트할 문서:

- `android_app_development_log.md`: 문제 원인, 의사결정, 검증 결과
- `android_app_build_notes.md`: 최신 APK, 빌드 방법, 버전별 요약
- `android_app_v0_2_release_report.md`: 릴리즈 단위 결과 보고
- `android_app_team_roles.md`: 팀 역할이나 개발 규칙이 바뀔 때
- `android_app_agent_coding_rules.md`: Codex 구현 기준이나 검증 원칙이 바뀔 때
- `android_app_map.md`: 화면, 데이터 흐름, 전략 계산, 저장소 구조가 바뀔 때
- `android_app_team_audit_report.md`: 팀별 감사 결과와 다음 개선 우선순위가 바뀔 때
- `signal_package_schema.md`: 신호 JSON 구조가 바뀔 때

## 기능 설계 전 체크 순서

새 기능이나 업데이트를 시작하기 전에 아래 순서로 검토한다.

1. Product 팀: 이 기능이 실전 운용 흐름에 필요한가.
2. 자산 운용 팀: 전략 계산이 검증된 원 규칙과 맞는가.
3. UX 팀: 사용자가 실제로 쉽게 입력하고 수정할 수 있는가.
4. Design 팀: 화면이 제품처럼 보이고 읽기 쉬운가.
5. 데이터 팀: 필요한 가격, 환율, 신호, 캐시 정책이 준비됐는가.
6. Android 개발 팀: 기존 저장 데이터와 업데이트 설치를 깨지 않는가.
7. QA 팀: 대표 시나리오로 계산과 화면을 검증했는가.
8. 문서 팀: 변경 이유와 결과를 MD에 기록했는가.

## Definition of Done

앱 업데이트는 아래 조건을 만족해야 완료로 본다.

- APK가 빌드된다.
- 최신 APK 경로가 문서에 기록된다.
- 변경된 전략 계산이 대표 금액 케이스로 검산된다.
- 새 화면이 하단 시스템 버튼, 상태바, 차트 영역과 겹치지 않는다.
- 기존 앱 위에 업데이트 설치했을 때 새 로직과 새 데이터가 적용된다.
- 사용자가 직접 지적한 문제뿐 아니라 같은 유형의 인접 문제도 함께 검토했다.
- 개발 로그에 문제, 원인, 수정, 검증 결과가 남아 있다.

## Codex 작업 규칙

앞으로 Codex는 Android 앱 관련 요청을 받을 때 다음 규칙을 따른다.

- `android_app_agent_coding_rules.md`를 Agent Coding Constitution으로 삼고 모든 작업 전에 먼저 적용한다.
- 해당 문서의 구현 전 사고, 단순성 우선, 외과적 변경, 목표 기반 실행 원칙을 함께 적용한다.
- 사용자가 화면 문제를 하나 지적하면 같은 화면의 입력, 계산, 데이터, 업데이트 문제까지 함께 점검한다.
- "일단 보이는 것만 수정"하지 않고 실전 투자 운용 목적에 맞는지 다시 확인한다.
- 전략 이름이 화면에 보인다고 전략이 구현된 것으로 보지 않는다.
- 추천 종목, 목표금액, 주문 수량, 검증 기준이 모두 연결되어야 전략 구현으로 인정한다.
- 앱 업데이트를 만들면 문서도 함께 업데이트한다.
