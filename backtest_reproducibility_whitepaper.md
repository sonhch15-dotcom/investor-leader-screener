# Backtest Reproducibility Whitepaper

작성일: 2026-07-10

## 1. 목적

백테스트는 같은 조건으로 다시 실행했을 때 같은 결과가 나와야 한다. 실행할 때마다 유니버스, 섹터 분류, 가격 데이터, 매매 규칙이 바뀌면 전략의 성과가 아니라 입력 데이터의 변화가 결과를 만든다.

이번 A/B/C 섹터 점수 변형 테스트에서 이 문제가 확인되었다. A안은 현재 산식과 같은 구조였지만, 기존 백테스트와 다른 유니버스/섹터 분류로 다시 실행되어 기존 공식 결과와 다른 종목을 선택했다. 따라서 해당 결과는 공식 전략 변경 근거가 아니라 재현성 관리 필요성을 보여준 1차 실험으로 기록한다.

## 2. 핵심 원칙

### 2.1 같은 백테스트는 같은 입력을 사용해야 한다

다음 항목이 하나라도 바뀌면 같은 전략 백테스트로 취급하지 않는다.

- 종목 유니버스
- 섹터/산업 분류
- 가격 데이터 소스
- 가격 데이터 기준일
- 조정주가 사용 여부
- 상장폐지/편입/제외 처리
- 월말 신호 확정일
- 매수일/매도일 캘린더
- 점수 산식
- 매수/매도 규칙
- 자금배분 규칙
- 수수료, 세금, 환율, 슬리피지 가정

### 2.2 유니버스와 섹터 분류는 반드시 고정한다

Leader2 전략은 섹터 리더십을 먼저 계산한다. 따라서 섹터 분류가 바뀌면 종목 선정도 바뀐다.

예를 들어 기존 백테스트에서 다음처럼 분리되던 섹터가:

```text
Electronic Components
Semiconductors
Software
```

새 실행에서 다음처럼 묶이면:

```text
Information Technology
```

리더 섹터 순위가 달라지고, 같은 산식이라도 선정 종목이 달라진다.

따라서 공식 백테스트는 실행 시점에 외부 사이트에서 섹터를 새로 가져오는 방식으로 하면 안 된다. 공식 검증에는 저장된 스냅샷 유니버스를 사용한다.

### 2.3 외부 데이터 자동 갱신 결과와 기존 결과를 직접 비교하지 않는다

다음 두 결과는 직접 비교하면 안 된다.

```text
기존 저장 백테스트 결과
vs
실행 시점에 유니버스/가격을 새로 받아 만든 백테스트 결과
```

외부 데이터가 바뀌면 차이가 생긴다. 이것은 전략 차이가 아니라 데이터 차이다.

### 2.4 한 번에 하나의 변수만 바꾼다

전략 개선 실험에서는 아래처럼 한 번에 하나만 바꾼다.

```text
고정: 유니버스, 섹터, 가격, 기간, 매매 규칙, 자금배분
변경: 개별 종목 점수 산식만 A/B/C
```

이 조건을 지키지 않으면 어떤 요인이 성과를 바꿨는지 판단할 수 없다.

## 3. 백테스트 등급

앞으로 모든 백테스트 결과는 등급을 붙인다.

### 3.1 Exploratory

목적: 아이디어 탐색

특징:

- 빠른 검토용
- 외부 데이터 재수집 가능
- 공식 전략 변경 근거로 사용 금지
- 결과가 좋아도 추가 검증 필요

### 3.2 Candidate

목적: 전략 후보 검증

필수 조건:

- 고정 유니버스 사용
- 고정 섹터 분류 사용
- 같은 기간/같은 가격 데이터 사용
- 기존 공식 전략과 동일한 매매 규칙으로 비교
- 변경 변수 명확히 기록

### 3.3 Official

목적: 실제 대시보드/운용 전략 반영

필수 조건:

- Candidate 검증 통과
- 자금 제한 계좌 시뮬레이션 통과
- 수수료/세금/환율/슬리피지 가정 기록
- 기존 공식 전략 대비 개선점 확인
- 최근 12개월 선정 종목과 매매 기록 검토
- 대시보드 반영 전 변경 사유 문서화

## 5. 백테스트 실행 체크리스트

새 백테스트를 실행하기 전에 다음을 확인한다.

- 기존 공식 결과와 비교하려는 테스트인가?
- 그렇다면 기존과 같은 유니버스를 쓰는가?
- 기존과 같은 섹터 분류를 쓰는가?
- 기존과 같은 가격 데이터 기간을 쓰는가?
- 기존과 같은 매수/매도 규칙을 쓰는가?
- 변경 변수는 하나인가?
- 결과 문서에 입력 데이터 버전이 기록되는가?
- 과거 결과를 덮어쓰기 전에 이전 결과를 보존하는가?
- 새 결과가 좋아도 자금 제한 계좌 검증을 별도로 통과했는가?
- 대시보드 반영 전 공식 전략 변경 사유가 문서화되었는가?

## 6. A/B/C 섹터 점수 변형 테스트에 대한 판정

`sector_score_variant_test.md`의 2026-07-10 결과는 다음 이유로 공식 비교가 아니다.

- 기존 `#backtest`와 동일한 유니버스가 고정되지 않았다.
- 기존 `#backtest`와 동일한 섹터 분류가 고정되지 않았다.
- 실행 시점의 외부 유니버스 구성이 반영되었다.
- A안의 산식은 현재 방식과 같지만, 입력 조건이 기존 공식 백테스트와 다르다.
- 따라서 기존 공식 `Cap27.5` 전략을 대체하거나 수정하는 근거로 사용하면 안 된다.

해당 결과는 `Exploratory` 등급으로만 보관한다.

## 7. 향후 수정 방향

A/B/C 점수 변형을 공식적으로 비교하려면 다음 순서로 다시 실행한다.

1. 기존 공식 백테스트와 동일한 유니버스 스냅샷을 고정한다.
2. 기존 공식 백테스트와 동일한 섹터 분류를 고정한다.
3. 기존 공식 백테스트와 동일한 가격 데이터 기간을 사용한다.
4. 매수/매도/자금배분 규칙은 기존 공식 전략과 동일하게 둔다.
5. 개별 종목 점수 산식만 A/B/C로 바꾼다.
6. 종목 선정력 테스트를 실행한다.
7. Cap27.5 자금배분과 6개월 50% 매도 + 주봉 연장 매도까지 포함한 계좌 검증을 실행한다.
8. 기존 공식 전략과 같은 표준 템플릿으로 비교한다.
9. 개선이 명확할 때만 Candidate로 승격한다.
10. Candidate 검증 후 대시보드 반영 여부를 결정한다.

## 8. 운영 원칙

앞으로 공식 전략과 비교하는 백테스트는 다음 원칙을 따른다.

```text
외부 데이터 재수집 결과를 기존 공식 결과와 직접 비교하지 않는다.
공식 비교는 항상 고정 스냅샷 기반으로 한다.
실험 결과가 좋아도 입력 조건이 다르면 공식 전략 변경 근거로 쓰지 않는다.
대시보드에 올리는 전략은 재현 가능한 백테스트만 사용한다.
```

이 원칙은 미국 주식, 한국 주식, 한국 ETF 전략 모두에 적용한다.

## 9. 2026-07-11 교정 백테스트 엔진 규칙

기존 미국 계좌 백테스트에서 다음 오류를 확인하고 수정했다.

- 각 매도 이벤트에 최종 평균 매도가를 반복 적용하던 문제
- 12개월 데이터가 없는 연장 보유분을 데이터 종료일에 강제 매도하던 문제
- 미청산 보유 종목을 매수가로 평가해 MDD를 축소하던 문제
- 매수 수수료가 포함된 평균 매수가를 사용하면서 계좌에서 수수료를 다시 차감할 수 있던 문제
- 가격 데이터를 다시 내려받을 때 과거 결과가 달라질 수 있던 문제

교정 엔진은 다음 규칙을 사용한다.

1. 매도 lot마다 실제 매도일, 매도가, 매도 비중을 저장한다.
2. 6개월 또는 12개월 조건이 아직 도래하지 않은 lot은 `right_censored`로 남긴다.
3. 미청산 보유분은 고정 가격 스냅샷의 주간 시장가로 평가한다.
4. 공식 위험 지표는 `maxDrawdown` 시장가 MDD를 사용한다.
5. `maxDrawdownAtCost`는 과거 결과 감사용으로만 보존하고 전략 비교에는 사용하지 않는다.
6. 최종 자산은 `현금 + 미청산 보유분 시장가`로 계산한다.
7. QQQ도 같은 시작일과 평가일의 시장가 곡선으로 비교한다.
8. 가격 스냅샷은 SHA-256 해시를 검증하며 해시가 다르면 같은 백테스트로 취급하지 않는다.

해시 정의:

- 유니버스 해시는 원본 파일 바이트가 아니라 정규화한 종목 배열의 `JSON.stringify` 결과에 대한 SHA-256이다.
- 가격 해시는 gzip 파일 바이트가 아니라 `version`, `firstDate`, `asOf`, `series`로 구성된 압축 전 정규 payload의 SHA-256이다.
- 따라서 `Get-FileHash`로 얻는 파일 해시와 아래 보고서의 재현성 해시는 서로 다르며, 전용 검증 스크립트가 정규 payload를 다시 계산해 비교한다.
- 19.8MB 가격 스냅샷은 저장소 재현성 자산으로 커밋하지만 GitHub Pages 배포물에서는 제외한다. 공개 JSON의 경로는 웹 URL이 아니라 저장소 상대 경로다.

관련 코드와 검증 명령:

```text
src/backtest-price-snapshot.mjs
src/backtest-execution-core.mjs
scripts/verify-corrected-backtest.mjs
npm run test:backtest-core
npm run verify:backtest -- --scale <scale.json> --lab <lab.json>
```

고정 스냅샷으로 A/C 전체 파이프라인을 재생하는 순서:

```text
node src/sector-score-variant-test.mjs --years 5 --universe data/universe.json --snapshot-in data/sector-score-price-snapshot-corrected-frozen-20260711.json.gz --output-suffix corrected-frozen-20260711
node src/scale-execution-test.mjs --source sector-score-variant-test-corrected-frozen-20260711.json --snapshot-in data/sector-score-price-snapshot-corrected-frozen-20260711.json.gz --strategy-key a_current_sector20 --output-suffix corrected-score-a-20260711
node src/scale-execution-test.mjs --source sector-score-variant-test-corrected-frozen-20260711.json --snapshot-in data/sector-score-price-snapshot-corrected-frozen-20260711.json.gz --strategy-key c_half_sector_normalized --output-suffix corrected-score-c-20260711
node src/strategy-development-lab.mjs --input scale-execution-test-corrected-score-a-20260711.json --output-suffix corrected-score-a-20260711
node src/strategy-development-lab.mjs --input scale-execution-test-corrected-score-c-20260711.json --output-suffix corrected-score-c-20260711

```

## 11. 시점별 구성 종목과 상장폐지 감사
