# Backtest Reproducibility Whitepaper

작성일: 2026-07-10

## 1. 목적

백테스트는 같은 조건으로 다시 실행했을 때 같은 결과가 나와야 한다. 실행할 때마다 유니버스, 섹터 분류, 가격 데이터, 매매 규칙이 바뀌면 전략의 성과가 아니라 입력 데이터의 변화가 결과를 만든다.

이번 A/B/C 섹터 점수 변형 테스트에서 이 문제가 확인되었다. A안은 현재 산식과 같은 구조였지만, 기존 백테스트와 다른 유니버스/섹터 분류로 다시 실행되어 기존 공식 결과와 다른 종목을 선택했다. 따라서 해당 결과는 공식 전략 변경 근거가 아니라 재현성 관리 필요성을 보여준 1차 실험으로 기록한다.

## 2. 핵심 원칙

### 2.1 같은 백테스트는 같은 입력을 사용해야 한다

다음 항목이 하나라도 바뀌면 같은 전략 백테스트로 취급하지 않는다.

- 종목 유니버스
- 섹터/산업 분류
- 가격 데이터 소스
- 가격 데이터 기준일
- 조정주가 사용 여부
- 상장폐지/편입/제외 처리
- 월말 신호 확정일
- 매수일/매도일 캘린더
- 점수 산식
- 매수/매도 규칙
- 자금배분 규칙
- 수수료, 세금, 환율, 슬리피지 가정

### 2.2 유니버스와 섹터 분류는 반드시 고정한다

Leader2 전략은 섹터 리더십을 먼저 계산한다. 따라서 섹터 분류가 바뀌면 종목 선정도 바뀐다.

예를 들어 기존 백테스트에서 다음처럼 분리되던 섹터가:

```text
Electronic Components
Semiconductors
Software
```

새 실행에서 다음처럼 묶이면:

```text
Information Technology
```

리더 섹터 순위가 달라지고, 같은 산식이라도 선정 종목이 달라진다.

따라서 공식 백테스트는 실행 시점에 외부 사이트에서 섹터를 새로 가져오는 방식으로 하면 안 된다. 공식 검증에는 저장된 스냅샷 유니버스를 사용한다.

### 2.3 외부 데이터 자동 갱신 결과와 기존 결과를 직접 비교하지 않는다

다음 두 결과는 직접 비교하면 안 된다.

```text
기존 저장 백테스트 결과
vs
실행 시점에 유니버스/가격을 새로 받아 만든 백테스트 결과
```

외부 데이터가 바뀌면 차이가 생긴다. 이것은 전략 차이가 아니라 데이터 차이다.

### 2.4 한 번에 하나의 변수만 바꾼다

전략 개선 실험에서는 아래처럼 한 번에 하나만 바꾼다.

```text
고정: 유니버스, 섹터, 가격, 기간, 매매 규칙, 자금배분
변경: 개별 종목 점수 산식만 A/B/C
```

이 조건을 지키지 않으면 어떤 요인이 성과를 바꿨는지 판단할 수 없다.

## 3. 백테스트 등급

앞으로 모든 백테스트 결과는 등급을 붙인다.

### 3.1 Exploratory

목적: 아이디어 탐색

특징:

- 빠른 검토용
- 외부 데이터 재수집 가능
- 공식 전략 변경 근거로 사용 금지
- 결과가 좋아도 추가 검증 필요

### 3.2 Candidate

목적: 전략 후보 검증

필수 조건:

- 고정 유니버스 사용
- 고정 섹터 분류 사용
- 같은 기간/같은 가격 데이터 사용
- 기존 공식 전략과 동일한 매매 규칙으로 비교
- 변경 변수 명확히 기록

### 3.3 Official

목적: 실제 대시보드/운용 전략 반영

필수 조건:

- Candidate 검증 통과
- 자금 제한 계좌 시뮬레이션 통과
- 수수료/세금/환율/슬리피지 가정 기록
- 기존 공식 전략 대비 개선점 확인
- 최근 12개월 선정 종목과 매매 기록 검토
- 대시보드 반영 전 변경 사유 문서화

## 4. 공식 백테스트 필수 기록 항목

백테스트 결과 문서에는 다음을 반드시 기록한다.

| 항목 | 필수 여부 | 설명 |
|---|---|---|
| run id | 필수 | 실행 결과를 식별하는 고유 ID |
| 생성일 | 필수 | 백테스트 생성 시각 |
| 데이터 기준일 | 필수 | 가격 데이터 마지막 날짜 |
| 유니버스 파일 | 필수 | 사용한 종목 목록 파일 |
| 유니버스 해시 | 필수 | 종목/섹터 변경 여부 확인용 |
| 섹터 분류 버전 | 필수 | 섹터 매핑 기준 |
| 가격 데이터 소스 | 필수 | Yahoo, 내부 캐시 등 |
| 가격 데이터 해시 | 권장 | 재현성 검증용 |
| 전략 ID | 필수 | 전략 식별자 |
| 점수 산식 버전 | 필수 | 개별 점수/섹터 점수 버전 |
| 매수 규칙 | 필수 | 월말 확정, 다음 거래일 매수 등 |
| 매도 규칙 | 필수 | 6개월 50%, 주봉 연장 등 |
| 자금배분 규칙 | 필수 | Cap27.5, 반복 추천 가중 등 |
| 비용 가정 | 필수 | 수수료, 세금, 슬리피지 |
| 벤치마크 | 필수 | QQQ, SPY, KOSPI200 등 |
| 변경 변수 | 필수 | 이전 공식 전략 대비 바뀐 점 |
| 결과 등급 | 필수 | Exploratory, Candidate, Official |

## 5. 백테스트 실행 체크리스트

새 백테스트를 실행하기 전에 다음을 확인한다.

- 기존 공식 결과와 비교하려는 테스트인가?
- 그렇다면 기존과 같은 유니버스를 쓰는가?
- 기존과 같은 섹터 분류를 쓰는가?
- 기존과 같은 가격 데이터 기간을 쓰는가?
- 기존과 같은 매수/매도 규칙을 쓰는가?
- 변경 변수는 하나인가?
- 결과 문서에 입력 데이터 버전이 기록되는가?
- 과거 결과를 덮어쓰기 전에 이전 결과를 보존하는가?
- 새 결과가 좋아도 자금 제한 계좌 검증을 별도로 통과했는가?
- 대시보드 반영 전 공식 전략 변경 사유가 문서화되었는가?

## 6. A/B/C 섹터 점수 변형 테스트에 대한 판정

`sector_score_variant_test.md`의 2026-07-10 결과는 다음 이유로 공식 비교가 아니다.

- 기존 `#backtest`와 동일한 유니버스가 고정되지 않았다.
- 기존 `#backtest`와 동일한 섹터 분류가 고정되지 않았다.
- 실행 시점의 외부 유니버스 구성이 반영되었다.
- A안의 산식은 현재 방식과 같지만, 입력 조건이 기존 공식 백테스트와 다르다.
- 따라서 기존 공식 `Cap27.5` 전략을 대체하거나 수정하는 근거로 사용하면 안 된다.

해당 결과는 `Exploratory` 등급으로만 보관한다.

## 7. 향후 수정 방향

A/B/C 점수 변형을 공식적으로 비교하려면 다음 순서로 다시 실행한다.

1. 기존 공식 백테스트와 동일한 유니버스 스냅샷을 고정한다.
2. 기존 공식 백테스트와 동일한 섹터 분류를 고정한다.
3. 기존 공식 백테스트와 동일한 가격 데이터 기간을 사용한다.
4. 매수/매도/자금배분 규칙은 기존 공식 전략과 동일하게 둔다.
5. 개별 종목 점수 산식만 A/B/C로 바꾼다.
6. 종목 선정력 테스트를 실행한다.
7. Cap27.5 자금배분과 6개월 50% 매도 + 주봉 연장 매도까지 포함한 계좌 검증을 실행한다.
8. 기존 공식 전략과 같은 표준 템플릿으로 비교한다.
9. 개선이 명확할 때만 Candidate로 승격한다.
10. Candidate 검증 후 대시보드 반영 여부를 결정한다.

## 8. 운영 원칙

앞으로 공식 전략과 비교하는 백테스트는 다음 원칙을 따른다.

```text
외부 데이터 재수집 결과를 기존 공식 결과와 직접 비교하지 않는다.
공식 비교는 항상 고정 스냅샷 기반으로 한다.
실험 결과가 좋아도 입력 조건이 다르면 공식 전략 변경 근거로 쓰지 않는다.
대시보드에 올리는 전략은 재현 가능한 백테스트만 사용한다.
```

이 원칙은 미국 주식, 한국 주식, 한국 ETF 전략 모두에 적용한다.
