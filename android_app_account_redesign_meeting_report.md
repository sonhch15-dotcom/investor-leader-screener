# Android App Account Redesign Meeting Report

작성일: 2026-07-09

## 1. 회의 결론

현재 Android 초안은 전략 신호를 앱 화면에 옮겨보는 데에는 의미가 있지만, 실제 운용 앱으로는 구조가 맞지 않는다. 가장 큰 문제는 `예수금`과 `보유 종목`이 앱 전체에 하나만 있는 것처럼 설계되어 있다는 점이다.

사용자가 설명한 실제 운용 흐름은 `android_app_real_usage_scenario.md`에 기준 사용자 여정으로 기록했다. 이 문서를 앞으로 화면 설계, 도메인 모델, QA 시나리오의 1차 기준으로 사용한다.

사용자는 실제로 아래 3개 계좌를 분리해서 운용한다.

| 계좌 | 기본 통화 | 기본 역할 | 기본 전략 연결 |
|---|---:|---|---|
| 달러 계좌 | USD | 미국 주식 운용 | US Leader2 전략 |
| 원화 계좌 | KRW | 한국 주식 운용 | KR Stock Leader2 전략 |
| 연금 계좌 | KRW | 한국 ETF 장기 운용 | KR ETF Core/Satellite 전략 |

따라서 앱의 핵심 원칙은 `계좌는 분리, 총자산은 통합`이다. 매수/매도 판단, 현금 부족, 수익률, lot, 알림은 계좌별로 계산하고, 자산 화면에서는 세 계좌를 합산해서 총자산과 변화 원인을 보여준다.

## 2. 팀별 결정

### 2.1 자산관리팀

현재 단일 예수금 모델은 폐기한다. 모든 돈의 흐름은 반드시 특정 계좌에 소속되어야 한다.

결정 사항:

- 입금, 출금, 환전, 매수, 매도, 배당, 수수료, 세금은 모두 `accountId`를 가진다.
- 달러 계좌의 USD 현금은 원화 계좌나 연금 계좌의 주문에 자동으로 사용하지 않는다.
- 원화 계좌와 연금 계좌는 둘 다 KRW를 쓰더라도 현금, 보유, 손익, 알림을 완전히 분리한다.
- 총자산은 모든 계좌를 KRW 기준으로 합산하고, 달러 계좌는 USD 원금/손익과 KRW 환산 손익을 함께 보여준다.
- 자산 변화는 아래 공식으로 분해한다.

```text
총자산 변화 = 투자 손익 + 환율 효과 + 입금/출금 + 배당 - 수수료 - 세금
```

필수 지표:

- 총자산 KRW
- 총자산 USD 환산
- 계좌별 평가액
- 계좌별 현금
- 계좌별 투자 중 금액
- 계좌별 실현손익
- 계좌별 미실현손익
- 달러 계좌의 환율 효과
- 연금 계좌의 목표 비중 이탈

### 2.2 UX팀

앱은 `종목 추천 앱`보다 `계좌별 투자 실행 콘솔`이어야 한다.

결정 사항:

- 첫 화면은 전체 총자산과 오늘 할 일을 함께 보여준다.
- 모든 액션 카드는 어느 계좌의 일인지 먼저 보여준다.
- 사용자는 `전체`, `달러`, `원화`, `연금`을 빠르게 전환할 수 있어야 한다.
- 주문 가이드는 계좌별 사용 가능 현금과 전략별 주문 한도를 함께 계산한다.
- 기록 화면의 첫 필드는 항상 계좌이며, 알림에서 진입하면 계좌가 자동 선택된다.
- 계좌 간 돈 이동은 `입출금`이 아니라 `이체/환전` 이벤트로 기록한다.
- 기록 누락 알림도 계좌별로 남긴다.

새 하단 내비게이션:

| 탭 | 목적 |
|---|---|
| 오늘 | 전체 Action Inbox와 계좌별 긴급 작업 |
| 자산 | 총자산, 계좌별 자산, 손익 분해 |
| 운용 | 계좌별 보유 종목, lot, 전략 상태 |
| 기록 | 계좌별 체결, 입출금, 환전, 배당 기록 |
| 설정 | 계좌, 전략, 데이터, 알림, 백업 |

`오늘` 화면의 우선순위:

1. 데이터 지연 또는 검증 필요
2. 계좌별 현금 부족
3. 오늘 처리해야 하는 매도/리밸런싱
4. 이번 달 신규 매수
5. 주문 후 기록 누락
6. 7일 이내 예정 이벤트

### 2.3 디자인팀

현재 초안의 문제는 기능보다 정보 위계와 화면 완성도다. 출시 제품 수준을 목표로 하려면 네이티브 디자인 시스템을 먼저 잡아야 한다.

결정 사항:

- Java View 기반 초안은 폐기하고 Kotlin + Jetpack Compose + Material 3로 재구현한다.
- 하단 내비게이션은 Compose `NavigationBar`와 `WindowInsets.navigationBars`를 사용해 Android 버튼과 절대 겹치지 않게 한다.
- 금액, 수량, 수익률은 카드 안에서 고정된 정렬 규칙을 가진다.
- 계좌별 색은 강한 원색이 아니라 작은 배지와 아이콘에만 쓴다.
- 카드 중첩을 금지하고, 반복 정보는 리스트/섹션으로 정리한다.
- 라이트 모드와 다크 모드를 모두 제공한다.
- 작은 화면, 큰 글자 모드, Android 3버튼 내비게이션에서 QA를 통과해야 한다.

권장 시각 언어:

- 배경: 밝은 중립 회색
- 카드: 흰색 또는 다크 표면
- 핵심 액션: 딥 블루
- 성공/상승: 사용자 설정 가능
- 위험/매도: 사용자 설정 가능
- 경고: 앰버
- 계좌 배지: 달러, 원화, 연금 3종의 낮은 채도 색상

### 2.4 개발팀

현재 앱은 SharedPreferences에 단일 장부를 저장한다. 출시 제품 수준으로 가려면 Room 기반 도메인 모델부터 다시 만들어야 한다.

- 서버/GitHub Actions는 공통 신호와 가격 데이터를 만든다.
- 앱은 계좌별 현금, lot, 보유 상태를 결합해 개인화된 주문 가이드를 만든다.
- 달러 계좌 전략 신호는 달러 계좌에만 적용한다.
- 한국 주식 신호는 원화 계좌에만 적용한다.
- 한국 ETF 리밸런싱은 기본적으로 연금 계좌에 적용한다.
- 같은 ETF를 원화 계좌에서도 운용하고 싶으면 계좌별 전략 연결을 추가 설정으로 허용한다.

## 3. 새 화면 구조

### 3.1 오늘

첫 화면은 전체 요약과 실행 작업을 동시에 보여준다.

상단:

- 총자산
- 전일/전월 대비 변화
- 데이터 기준일
- 마지막 동기화

계좌 요약:

- 달러 계좌: USD 평가액, KRW 환산, 현금
- 원화 계좌: KRW 평가액, 현금
- 연금 계좌: KRW 평가액, 현금, 목표 비중 이탈

Action Inbox:

- `[달러] TECH 신규 매수`
- `[원화] 삼성전기 신규 매수`
- `[연금] ETF 리밸런싱`
- `[달러] STX 6개월 50% 매도`
- `[원화] 매수 기록 누락`

### 3.2 자산

자산 화면은 `전체`를 기본으로 열고, 계좌별 상세로 내려간다.

구성:

- 총자산 카드
- 계좌별 자산 카드 3개
- 손익 분해 카드
- 환율 효과 카드
- 자산 배분 차트
- 계좌별 현금/투자 중 자산

중요한 UX:

- 총자산은 KRW 기준을 기본으로 표시한다.
- 달러 계좌에는 USD 기준 손익과 KRW 환산 손익을 같이 표시한다.
- 연금 계좌는 일반 계좌와 섞지 않고 별도 카드로 유지한다.

### 3.3 운용

운용 화면은 계좌 세그먼트를 중심으로 한다.

필터:

- 전체
- 달러
- 원화
- 연금

계좌별 표시:

- 달러: 미국 주식 lot, 6개월/12개월 일정, 주봉 훼손
- 원화: 한국 주식 lot, 가격 제한폭/거래정지 주의
- 연금: ETF 목표 비중, 현재 비중, 리밸런싱 필요 금액

### 3.4 기록

기록은 계좌 중심 입력으로 바꾼다.

빠른 액션:

- 매수 기록
- 매도 기록
- 입금
- 출금
- 환전/이체
- 배당
- 수수료/세금

입력 원칙:

- 계좌 선택이 먼저다.
- 액션에서 들어오면 계좌, 종목, 전략, 예상 금액이 자동 입력된다.
- 사용자는 실제 체결 수량, 체결가, 수수료, 세금만 확인한다.
- 환전은 달러 계좌와 원화 계좌 사이의 현금 이동으로 기록한다.

### 3.5 설정

설정은 앱 완성도에 직접 영향을 주는 항목만 둔다.

- 계좌 관리
- 계좌별 전략 연결
- 수수료/세금 기본값
- 환율/가격 데이터 상태
- 알림 정책
- 금액 숨김 기본값
- 라이트/다크 모드
- 상승/하락 색상 관습
- 백업/복원

## 4. 하단 Android 버튼 겹침 이슈

현재 초안 앱의 겹침 원인은 `targetSdk 36` 환경에서 시스템 내비게이션 영역을 피하지 않는 커스텀 하단바를 사용했기 때문이다.

임시 대응:

- Java 초안 앱에 시스템 navigation bar bottom inset을 하단바 높이와 padding에 반영한다.

최종 대응:

- Compose `Scaffold`
- Material 3 `NavigationBar`
- `WindowInsets.safeDrawing`
- `navigationBarsPadding`
- 작은 화면, 큰 글자, 3버튼 내비게이션, 제스처 내비게이션 스크린샷 QA

완료 기준:

- Android 하단 버튼과 앱 하단 내비게이션이 겹치지 않는다.
- 마지막 리스트 아이템과 주요 버튼이 시스템 내비게이션에 가려지지 않는다.
- 모든 바텀시트의 확인 버튼이 안전 영역 위에 위치한다.

## 5. 새 개발 단계

### Phase 0. 현재 초안 동결

목표:

- 현재 Java APK는 전략/데이터 연동 검증용으로만 유지한다.
- 출시 제품 수준 앱은 Compose/Room 기반으로 새로 만든다.

완료 기준:

- 현재 초안의 한계가 문서화된다.
- 단일 예수금 모델을 더 확장하지 않는다.

### Phase 1. 계좌 중심 도메인 모델

작업:

- Room 스키마 설계
- Account, CashBalance, Transaction, Lot, Snapshot 구현
- 3개 기본 계좌 생성 플로우
- 계좌별 현금과 보유 계산 테스트

완료 기준:

- 달러/원화/연금 계좌의 현금과 보유가 완전히 분리된다.
- 총자산만 통합 계산된다.

### Phase 2. UX/디자인 시스템 재설계

작업:

- Compose Material 3 테마
- 하단 내비게이션 안전 영역 처리
- 계좌 세그먼트
- 자산 카드
- ActionCard
- OrderGuideSheet
- ExecutionRecordSheet

완료 기준:

- 작은 화면과 Android 하단 버튼 환경에서 겹침이 없다.
- 금액/수량/수익률이 한눈에 읽힌다.

### Phase 3. 계좌별 주문 가이드

작업:

- 달러 계좌 미국 주식 주문 가이드
- 원화 계좌 한국 주식 주문 가이드
- 연금 계좌 ETF 리밸런싱 가이드
- 현금 부족, 한도 초과, 데이터 지연 상태 처리

완료 기준:

- 어느 계좌의 현금으로 어떤 종목을 사야 하는지 명확하다.
- 다른 계좌 현금을 실수로 끌어다 쓰지 않는다.

### Phase 4. 자산관리 화면

작업:

- 총자산
- 계좌별 자산
- 투자 손익
- 환율 효과
- 입출금
- 배당/수수료/세금
- 월별 자산 스냅샷

완료 기준:

- 사용자가 자산이 왜 변했는지 계좌별/전체 기준으로 이해할 수 있다.

### Phase 5. 알림과 Action Inbox

작업:

- 계좌별 전략 알림
- 매수/매도/리밸런싱/기록 누락 상태
- WorkManager 주기 동기화
- 알림 클릭 시 해당 계좌의 기록 화면으로 이동

완료 기준:

- 알림이 어떤 계좌의 어떤 행동인지 명확하다.
- 알림 권한이 꺼져 있어도 앱 안 Action Inbox에서 놓치지 않는다.

### Phase 6. QA와 제품 완성도

작업:

- 3버튼 내비게이션 QA
- 제스처 내비게이션 QA
- 작은 화면 QA
- 큰 글자 모드 QA
- 다크 모드 QA
- 데이터 지연/오류 상태 QA
- Room 백업/복원 QA

완료 기준:

- 개인용 앱이지만 내부 출시 후보 수준의 완성도를 가진다.

## 6. MVP 재정의

새 MVP는 `추천 종목 표시 앱`이 아니라 `3계좌 전략 운용 앱`이다.

반드시 포함:

1. 달러, 원화, 연금 3개 계좌 기본 생성
2. 계좌별 현금 입력
3. 계좌별 입금, 출금, 환전/이체 기록
4. 계좌별 매수/매도 기록
5. 계좌별 lot 관리
6. 계좌별 전략 연결
7. 계좌별 주문 가이드
8. 전체 총자산 통합 표시
9. 계좌별 자산 표시

## 8. 최종 권고

현재 앱을 조금씩 고치는 방식은 금방 한계가 온다. 단일 장부를 3계좌 장부로 억지 확장하면 자산 계산, 알림, lot, 환율, 기록 화면이 계속 꼬인다.

따라서 출시 제품 수준을 목표로 한다면 다음 단계는 명확하다.

1. 현재 Java 초안은 데이터 연동 검증용으로만 둔다.
2. 실제 앱은 Compose + Room으로 새로 만든다.
3. 첫 개발 단위는 화면이 아니라 `계좌 중심 장부 엔진`으로 잡는다.
4. 그 위에 오늘, 자산, 운용, 기록, 설정 화면을 다시 올린다.
5. Android 시스템 안전 영역과 접근성은 디자인 QA의 필수 게이트로 둔다.
