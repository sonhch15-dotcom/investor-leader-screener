## 결론

현재 앱은 단순한 웹 대시보드 복제 수준은 넘어섰다. 3계좌 구조, 시장별 운용 분리, 총자산 원화 통합, 월간 신호와 주문 가이드, ETF 목표 비중, 주봉 훼손 감시, 가격/환율 동기화 골격은 들어가 있다.

하지만 아직 실전 투자 운용 앱으로 쓰기에는 P0 리스크가 남아 있다. 핵심은 세 가지다.

1. 장부가 틀어졌을 때 복구할 방법이 부족하다.
2. 일부 조건에서 잘못된 주문 가이드가 나올 수 있다.
3. 지연/실패 가격과 환율이 주문 가이드에 그대로 쓰일 수 있다.

따라서 다음 APK는 새 화면을 늘리기보다 `장부 무결성`, `주문 가이드 차단 조건`, `ETF 체결 기록 루프`, `데이터 신뢰도 표시`를 먼저 고쳐야 한다.

### Product / 사용자 운용 팀

판단:

- 앱의 방향은 이제 단순 대시보드가 아니라 실전 운용 보조 도구에 가까워졌다.
- 미국/한국/연금 ETF 분리 운용, 총자산 원화 통합, 주문 가이드, 체결 기록, 장부 점검의 골격은 목적에 맞다.
- 사용자가 실제로 혼란을 겪은 `투자 손익은 마이너스인데 자산 변화는 플러스` 문제를 원금 기준으로 재정의한 것은 올바른 방향이다.

남은 위험:

- 아직 사용자가 실제 증권사 잔고와 앱 장부를 주기적으로 맞추는 흐름이 충분히 자동화되어 있지 않다.
- `장부 점검`은 문제 발견까지는 하지만, 문제 해결 동선은 아직 수동 정정에 많이 의존한다.

결론:

- 다음 우선순위는 새 기능 확장보다 `장부 점검 -> 정정 -> 재계산 확인` 흐름을 더 매끄럽게 만드는 것이다.

### UX 팀

판단:

- 기록 타임라인 기본 접힘, 최근 기록 요약, 장부 점검 카드는 사용 피로를 낮춘다.
- 자산 탭의 `원금 대비 / 투자 손익 / 환율 영향` 3칸 요약은 기존 상세 숫자 나열보다 이해하기 쉽다.
- USD 입출금 정정에서 당시 환율을 넣을 수 있게 된 점은 실제 문제 해결에 직접 연결된다.

남은 위험:

- 장부 점검 카드의 항목이 늘어나면 기록 탭이 다시 길어질 수 있다.
- 환율 누락 기록이 여러 건일 때 사용자가 어떤 기록부터 정정해야 할지 우선순위가 부족하다.
- 매도 정정은 아직 취소 후 재입력 방식이라 실사용 중 피로가 남는다.

결론:

- 다음 UX 개선은 `문제 있는 기록만 보기`, `장부 점검 항목별 바로가기`, `매도 정정 UI`가 적절하다.

### Design / 시각 시스템 팀

판단:

- 버튼 크기와 카드 밀도는 초기 버전보다 안정됐다.
- 자산 요약 3칸은 시각적 위계를 만드는 데 도움이 된다.
- 도넛/라인 차트는 기능적으로 들어갔지만, 출시 제품 수준으로 보려면 차트 레이블과 빈 상태 디자인이 더 다듬어져야 한다.

남은 위험:

- 네이티브 Java 프로그램식 UI라 컴포넌트 재사용과 시각 일관성 관리가 점점 어려워지고 있다.
- 숫자와 경고가 많은 화면에서 색상만으로 의미를 구분하면 피로해질 수 있다.
- 작은 화면에서 카드가 계속 길어지는 문제가 아직 남아 있다.

결론:

- 다음 디자인 개선은 `상태 카드`, `요약 카드`, `문제 카드`의 공통 컴포넌트화와 빈 상태 UI 정리다.

### 자산 운용 / 전략 검증 팀

판단:

- 미국 Cap27.5, 한국 Leader2, ETF 비중 계산은 이전 오류를 지나 현재는 핵심 규칙이 반영되어 있다.
- 실제 사례 self-test가 들어가면서 원금/환율/손익 계산의 회귀 방지 장치가 생겼다.
- 평균원가 임시평가 정책은 체결 직후 stale quote로 잘못된 손익을 보여주는 문제를 줄인다.

남은 위험:

- 입금 당시 환율이 없는 기존 기록은 여전히 추정값이다.
- 환전 이벤트가 많은 경우 `USD 순입금 환율 영향`만으로는 정밀한 환전 손익을 설명하기 어렵다.
- 배당, 세금, 수수료, 환전 손익을 완전 분해하는 회계 모델은 아직 없다.

결론:

- 다음 전략/자산 작업은 `정밀 손익 분해`보다 먼저 `원금 기준 계산의 표시 신뢰성`과 `증권사 대조 후 차이 해결`을 고도화해야 한다.

### 데이터 / 자동화 팀

판단:

- GitHub Pages 신호 패키지와 앱 직접 시세 overlay 구조는 무료 운용 목적에 맞다.
- 앱 foreground 복귀와 주요 화면 진입 시 15분 throttle 자동 동기화는 개인용 앱에 적절하다.
- 가격/환율 실패가 장부 점검과 데이터 동기화 카드에 노출되는 점은 좋다.

남은 위험:

- Yahoo/Frankfurter는 공식 보장 API가 아니므로 실패나 차단 가능성이 있다.
- 한국 종목/ETF 시세 fallback의 안정성은 실제 사용 데이터가 더 쌓여야 판단 가능하다.
- 직접 시세 실패 메시지가 아직 길고 기술적으로 보여질 수 있다.

결론:

- 다음 데이터 작업은 `실패 메시지 사용자화`, `대체 소스 후보 설계`, `실패해도 마지막 정상 시세로 안전하게 표시` 정책이다.

### Android 개발 팀

판단:

- `0.3.27`은 versionCode/versionName/APK 파일명이 일치한다.
- SharedPreferences 기반 장부가 기능적으로는 충분히 동작하지만, 기능이 늘면서 저장소 책임이 커지고 있다.
- `MainActivity`가 화면과 로직을 많이 안고 있어 유지보수 리스크가 커지고 있다.

남은 위험:

- Room/Repository/ViewModel 같은 구조가 없어 화면 변경과 계산 변경이 결합되어 있다.
- 백업은 클립보드 JSON 중심이라 장기 운용 데이터 보호에는 부족하다.
- 업데이트 설치 후 실제 사용자 데이터 migration 검증은 더 체계화해야 한다.

결론:

- 당장 Compose 전환보다 먼저 `계산 로직 분리`, `장부 진단 로직 분리`, `백업 파일 저장`이 현실적인 다음 단계다.

### QA / 검증 팀

판단:

- `StrategyMathSelfTest`, `assembleDebug`, `aapt dump badging`, `git diff --check`는 통과했다.
- 실제 사용자 숫자 사례가 self-test에 들어간 점은 매우 중요하다.

남은 위험:

- UI 스크린샷 기반 회귀 테스트는 아직 없다.
- 업데이트 설치 후 기존 장부가 유지되는지, 환율 누락 기록 정정 후 자산 요약이 바뀌는지 자동 검증은 없다.
- 매도 lot, ETF 리밸런싱, 한국 주식 주문 흐름의 통합 시나리오 테스트가 더 필요하다.

결론:

- 다음 QA 우선순위는 `사용자 실제 시나리오 5개`를 self-test/수동 체크리스트로 고정하는 것이다.

## P0: 다음 APK 전 필수 개선

### 1. 장부 입력 검증과 복구 안전장치

문제:

- 매수/매도/입출금 입력값이 저장소 레벨에서 검증되지 않는다.
- 파싱 실패 시 `0`으로 저장될 수 있다.
- 초과 매도 시 현금은 입력 수량 전체 기준으로 증가하지만 보유 lot 차감은 보유 수량으로 제한되어 현금이 부풀 수 있다.
- 기록 수정/삭제/되돌리기가 없다.
- `entries_v2` JSON이 깨지면 빈 배열로 처리되고, 이후 새 기록 저장 시 기존 장부를 덮을 수 있다.

근거:

- `MainActivity.java`: `showTradeDialog`, `showManualSellDialog`, `showCashDialog`, `number`
- `LedgerStore.java`: `recordTrade`, `addCashEvent`, `holdings`, `append`, `readEntries`
- `android_app_map.md`: 장부 백업/복원과 수정/삭제가 백로그로 표시됨

개선 방향:

- `LedgerStore`에 저장 전 검증 API를 둔다.
- 수량, 가격, 수수료, 금액은 음수/0/빈 값 저장을 막는다.
- 매도는 보유 수량 초과 저장을 막는다.
- 매수는 해당 통화 현금 부족을 명시적으로 경고한다.
- JSON 파싱 실패 시 원문을 `corrupt_entries_*`로 보존하고 쓰기를 차단한다.
- 최소 기능으로라도 기록 삭제/되돌리기 또는 정정 기록을 제공한다.
- 장기적으로 Room + migration + JSON 백업/복원으로 이전한다.

### 2. 목표금액보다 1주 가격이 큰 경우의 주문 가이드 차단

문제:

- 목표 금액이 1주 가격보다 작거나 목표가 `0`이어도 `Math.max(1, floor(target / reference))` 때문에 1주 매수가 기본 제안될 수 있다.
- 현재 한국 Leader2 예시에서 종목당 목표 150만원인데 고가주 기준가가 200만원을 넘으면 목표보다 큰 매수가 제안될 수 있다.

근거:

- `MainActivity.java`: `showOrderGuide`, `showTradeDialog`, `validationText`
- `signals/latest.json`: 한국 주식 고가주 기준가

개선 방향:

- `target <= 0`이면 주문 수량을 `0` 또는 `보류`로 표시한다.
- `target < referencePrice`이면 자동 1주 제안 대신 `1주가 목표금액 초과` 상태를 표시한다.
- 사용자가 명시적으로 1주 매수를 선택할 때만 진행하게 한다.
- `현실적 완료` 판정과 `주문 추천` 판정을 분리한다.

### 3. 지연/실패 가격과 환율의 주문 사용 차단

문제:

- 현재 내장 가격/환율 상태가 `delayed` 또는 fallback이어도 주문 가이드가 계속 생성된다.
- quote별 `status`, `priceDate`, `source`가 주문 차단 조건으로 쓰이지 않는다.
- 환율 상태가 별도 파싱되지 않는다.

근거:

- `prices/latest.json`: 일부 quote가 fallback/failed/delayed 상태
- `fx/latest.json`: manual-default 환율
- `SignalRepository.java`: `referencePrice`
- `MainActivity.java`: `addDataStatus`, `buildActions`, `showOrderGuide`

개선 방향:

- 가격/환율 상태가 `normal`이 아니면 신규 매수/리밸런싱 버튼을 비활성화한다.
- 사용자가 강행할 수 있더라도 `데이터 검증 필요` 단계를 먼저 통과하게 한다.
- `fxStatus`, `quoteFailedCount`, `staleQuoteCount`, `lastSyncError`, `lastSuccessfulSyncAt`을 저장하고 화면에 표시한다.
- GitHub Actions에서 핵심 quote/FX freshness check 실패 시 Pages 배포를 중단하거나 `needs_review`로 배포한다.

### 4. ETF 리밸런싱을 체결 기록으로 연결

문제:

- ETF 리밸런싱 다이얼로그는 목표/현재/매수·매도 수량을 계산하지만 `확인`만 있고 체결 기록으로 이어지지 않는다.

근거:

개선 방향:

## P1: 실전 운용 신뢰도 개선

### 1. 통화와 계좌 경계 강화

문제:

- 모든 계좌에서 KRW/USD 입출금이 가능하다.
- 미국 계좌에서 USD 현금이 0이면 KRW 보조 현금을 USD 주문 가능액처럼 환산할 수 있다.
- 체결 기록은 신호 통화 현금만 차감하므로 환전 없이 통화가 섞일 수 있다.

개선 방향:

- 계좌별 허용 통화를 둔다.
- 한국 주식/ETF 계좌는 기본적으로 KRW만 허용한다.
- 미국 계좌의 KRW는 `보조 현금`이 아니라 `환전 대기 현금`으로 구분한다.
- FX 환전 이벤트를 장부 타입으로 추가한다.

### 2. Action Inbox 상태 저장

문제:

- `보류` 버튼은 Toast만 띄우고 상태를 저장하지 않는다.
- 이미 처리한 액션과 보류한 액션이 계속 다시 생성될 수 있다.

개선 방향:

- ActionItem에 `pending`, `done`, `snoozed`, `needs_record`, `blocked_data` 상태를 둔다.
- 보류는 날짜/사유와 함께 저장한다.
- 체결 기록 저장 후 해당 action은 완료 상태가 되어야 한다.

### 3. lot 단위 운용 모델

반영 상태:

- `0.3.6`에서 장부 기반 FIFO lot 계산 모델과 주식 계좌 lot 이벤트 UI를 1차 반영했다.
- `0.3.7`에서 특정 lot 선택 매도와 매도 기록의 실현손익 저장을 1차 반영했다.
- 매수 기록은 open lot으로 파생되고, 매도 기록은 같은 종목 lot을 FIFO 순서로 차감한다.
- 운용 화면은 6개월 50% 매도, 12개월 전량 매도, 주봉 훼손 잔여 매도를 lot별 액션으로 표시한다.
- 사용자가 특정 lot을 선택하면 해당 lot을 먼저 차감하고, 선택하지 않으면 FIFO 자동으로 처리한다.
- 보유 종목의 평균 원가와 cap 판단은 남은 open lot의 잔여 원가 기준으로 집계한다.
- `0.3.8`에서 실현손익, 미실현손익, 투자 손익 합계를 자산 화면에 집계했다.
- `0.3.10`에서 손익 추세 그래프를 추가했다.

남은 문제:

- 배당, 세금, 환율효과는 아직 독립 분해하지 않는다.
- lot 선택 UI는 버튼 목록 기반이라 lot이 많아질 때 사용성이 떨어질 수 있다.
- 현재 lot은 `entries_v2`에서 파생 계산하므로 장기적으로 Room migration 시 별도 lot/transaction projection을 설계해야 한다.

개선 방향:

- 배당, 세금, 환율효과를 별도 장부 이벤트와 환율 히스토리로 분해한다.
- lot 선택 UI를 접기/상세/검색 구조로 개선한다.
- 전략 엔진을 순수 로직으로 분리하고 lot 이벤트 계산 테스트를 추가한다.

### 4. 전략 계산의 원천 데이터 보존

문제:

- `orderHint`, `warnings`, `validFrom`이 앱 모델에 보존되지 않는다.
- ETF drift threshold가 패키지의 5%와 코드의 1.5%로 다르다.
- Cap 상한은 누적 원금이 아니라 잔여 원가 기준으로 열릴 수 있다.
- 초기 3개월 판정이 달력 월이 아니라 매수 기록 수로 계산된다.

개선 방향:

- `StrategySignal`에 `orderHint`, `warnings`, `validFrom`을 파싱한다.
- 전략 계산은 코드 상수보다 신호 패키지 정책을 우선한다.
- cap 판단은 `holding.cost`가 아니라 누적 원금 또는 active lot 원금 정책을 명확히 사용한다.
- ramp 기간은 기록 개수가 아니라 계좌/전략 시작월 또는 월차 기준으로 계산한다.

### 5. 원격 패키지 검증 강화

문제:

- 원격 패키지는 `packageVersion` 비교만 하고 manifest hash, schema major, 필수 파일 status를 검증하지 않는다.
- manifest에 선언된 파일과 APK asset에 포함되는 파일 세트가 다르다.
- 자동 동기화 실패가 non-interactive 상황에서는 사용자에게 남지 않는다.

개선 방향:

- remote fetch 후 sha256, schema major, required file, file status를 검증한다.
- 검증 후에만 all-or-nothing으로 cache를 교체한다.
- 앱 asset manifest와 Pages manifest를 분리하거나 `runtimeRequired`를 둔다.
- 동기화 실패와 마지막 성공 시간을 기록 탭과 오늘 카드에 표시한다.

## P2: 제품 완성도 개선

### 1. 기록 UX 개선

- `현재 추천 매수 기록`은 첫 buy 신호가 아니라 계좌 선택 -> 종목 선택 -> 체결 기록 순서로 바꾼다.
- 기록 탭에서 입금, 출금, 매수, 매도, ETF 리밸런싱 기록을 모두 시작할 수 있게 한다.
- 체결가/수수료 입력에도 금액 포맷과 빠른 입력 컨트롤을 제공한다.

### 2. 연구 전략 선택 UX

- 신호 패키지가 없는 연구 전략은 선택 불가 또는 확인 경고를 둔다.
- 연구 전략 선택 시 현재 실행 신호가 사라질 수 있음을 명확히 알린다.

### 3. 디자인 시스템 정리

- 하단 내비게이션은 아이콘+라벨, active indicator, top divider를 적용한다.
- 도넛 차트는 stroke outer bounds 기준으로 clipping을 다시 계산한다.
- 카드당 주 CTA 하나만 강조하고 보조 액션은 작은 버튼/overflow로 낮춘다.
- 주문/ETF 가이드는 `AlertDialog.setMessage()` 문자열이 아니라 표/요약 블록/CTA 구조로 만든다.
- 보유 종목 차트는 Top 5 + 기타 묶음을 도입한다.

- 투자손익, 환율효과, 입출금, 수수료/세금 분해를 계산한다.

## v0.3.11 추가 감사 결과: 월간 목표와 체결 기준 분리

사용자 STX 실전 매수 사례를 검토한 결과, 앱이 남은 현금으로 다시 계산한 추가 주문 가능액을 누적 보유 원금과 비교해 초과로 오판할 수 있었다.

결론:

- 전략 자체의 Cap27.5 적용보다 앱의 검증 기준 분리가 문제였다.
- 실전 운용 화면에서는 `이번 목표`, `이번 체결`, `남은 매수`, `총 보유 원금`, `종목 한도`를 분리해야 한다.
- 매수 후 줄어든 현금은 추가 주문 가능액 계산에만 쓰고, 이미 확정된 월간 목표를 축소하는 근거로 쓰면 안 된다.

반영 상태:

- `OrderPlan` 계산으로 이번 목표와 이번 체결을 분리했다.
- Action Inbox, 운용 카드, 주문 가이드, 체결 저장 후 토스트가 같은 기준을 사용한다.
- `signalId`가 있는 기록은 해당 신호 체결로 직접 합산하고, 구버전 기록은 종목/전략/유효기간으로 보정한다.

## v0.3.12 추가 감사 결과: 이번 체결과 리밸런싱 의미 분리

0.3.11 적용 후 STX 기록을 다시 검토한 결과, 이번 신호 기간의 매수 총액을 `이번 체결`로 더하면서 중간 매도 원가를 차감하지 않는 문제가 남아 있었다.

결론:

- 주식 매수 신호의 `이번 체결`은 매수 총액이 아니라 현재 남아 있는 이번 신호 lot 원가여야 한다.
- ETF 리밸런싱과 달리 주식 매수 신호는 목표 이상 매수 자체를 즉시 조정 필요로 보지 않는다.
- 조정 필요 초과는 종목 한도 초과처럼 전략상 실제 위험 한도를 넘을 때만 표시한다.

반영 상태:

- `HoldingLot`에 `signalId`, `strategyKey`를 보존해 이번 신호 lot을 식별한다.
- `OrderPlan.executedOrderValue`는 open lot 잔여 원가 기준으로 계산한다.
- 주식 매수 검증은 목표 이상/한도 이내를 완료로 처리한다.

## v0.3.13 추가 감사 결과: 목표 범위와 한도 이내 분리

사용자 지적대로 목표 10,000 USD에 12,000 USD를 산 것은 목표 범위가 아니다.

결론:

- `목표 범위`는 목표 대비 허용 오차 안일 때만 써야 한다.
- 목표를 명확히 넘었지만 종목 한도 안인 경우는 `확인 필요` 상태다.
- 종목 한도까지 넘은 경우만 `조정 필요` 상태다.

반영 상태:

- 주식 매수 신호의 초과 상태 문구를 `확인 필요: 목표보다 N 초과 · 한도 이내`로 변경했다.
- 목표 범위, 목표 초과, 종목 한도 초과의 의미를 분리했다.
