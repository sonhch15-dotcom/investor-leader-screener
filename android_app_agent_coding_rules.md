# Android App Agent Coding Rules

작성일: 2026-07-09

이 문서는 Investor Run Android 앱을 Codex로 개발할 때 따를 코딩 기준이며, 이 프로젝트의 Agent Coding Constitution으로 지정한다.
기준은 `multica-ai/andrej-karpathy-skills` 저장소의 `CLAUDE.md`를 참고했다.

주의: 해당 저장소는 Andrej Karpathy 본인이 직접 작성한 공식 저장소로 확인된 것은 아니며, 저장소 설명상 Karpathy의 LLM 코딩 관찰에서 파생된 가이드라인이다. 따라서 이 문서는 "Karpathy-inspired" 기준으로 사용한다.

## 핵심 원칙

## 헌법 지위

이 문서는 단순 참고 자료가 아니라 Codex가 모든 작업 전에 먼저 떠올려야 하는 상위 규칙이다.

- 앱 설계, 코드 수정, APK 빌드, 문서 업데이트를 시작하기 전에 이 문서의 원칙을 먼저 적용한다.
- 사용자 요청, 팀 역할 문서, 기존 구현 방식이 충돌할 때는 이 문서의 "추측하지 않기", "단순성", "외과적 변경", "검증 가능한 목표" 원칙을 우선한다.
- 단, 사용자의 명시적 최신 요청이 있으면 그 요청을 최우선으로 해석하되, 구현 방식은 이 문서의 원칙 안에서 정한다.
- 작업 결과를 보고할 때는 필요한 경우 어떤 원칙을 적용했는지 함께 설명한다.
- 이 헌법을 바꾸려면 사용자의 명시적 요청과 문서 업데이트가 필요하다.

### 1. 구현 전 사고

추측으로 바로 구현하지 않는다.

- 요구가 애매하면 먼저 가정을 분명히 적는다.
- 해석이 여러 개라면 임의로 고르지 않고 선택지를 드러낸다.
- 더 단순한 방법이 있으면 말한다.
- 전략 계산, 자산 배분, 통화 처리처럼 틀리면 실제 투자 판단에 영향을 주는 부분은 혼란을 숨기지 않는다.

Investor Run 적용:

- "미국/한국/ETF 전략이 들어갔는가"라는 질문에는 코드와 신호 패키지를 직접 확인한 뒤 답한다.
- 전략명만 표시되는 것은 구현으로 보지 않는다. 추천 종목, 목표 금액, 현재 보유, 주문 수량, 검증 기준이 모두 연결되어야 전략 구현으로 인정한다.
- 사용자 계좌 구조, 통화, 운용 단위가 불명확하면 기능 구현 전에 흐름을 재확인한다.

### 2. 단순성 우선

요청을 해결하는 최소 구조로 만든다.

- 요청되지 않은 기능을 넣지 않는다.
- 한 번만 쓰는 코드를 위해 추상화를 만들지 않는다.
- 미래 확장성을 핑계로 UI와 로직을 복잡하게 만들지 않는다.
- 200줄로 쓴 코드가 50줄로 가능하면 줄인다.

Investor Run 적용:

- 버튼, 입력창, 카드가 커지고 많아지는 방향보다 실제 매수/매도 기록이 빨라지는 방향을 우선한다.
- "전략 잠시 중지"처럼 목적이 불명확한 버튼은 기본 기능으로 넣지 않는다.
- 개인 사용 앱이어도 출시 제품 수준의 품질을 목표로 하되, 서버/인프라는 무료 운영 가능성을 해치지 않는 선에서 단순하게 유지한다.

### 3. 외과적 변경

필요한 곳만 고친다.

- 주변 코드, 주석, 포맷을 이유 없이 정리하지 않는다.
- 기존 동작을 이해하지 못한 채 리팩터링하지 않는다.
- 기존 스타일을 따른다.
- 내가 만든 변경으로 생긴 미사용 코드만 정리한다.
- 기존에 있던 죽은 코드는 발견해도 임의 삭제하지 않고 보고한다.

Investor Run 적용:

- 전략 로직을 수정할 때 UI까지 무리하게 바꾸지 않는다. 반대로 UI 수정 중 전략 계산식을 건드리지 않는다.
- `LedgerStore`, `SignalRepository`, `MainActivity`, asset JSON의 책임을 섞지 않는다.
- 기존 사용자 저장 데이터가 있는 상황을 항상 전제로 한다.

### 4. 목표 기반 실행

"작동하게 해줘"를 검증 가능한 목표로 바꾼다.

- 버그 수정은 재현 조건과 통과 기준을 함께 둔다.
- 전략 계산 수정은 대표 금액 케이스로 검증한다.
- APK 업데이트는 설치 후 캐시/원격 동기화까지 확인한다.
- 다단계 작업은 짧은 계획과 각 단계별 검증 기준을 둔다.

Investor Run 필수 검증 예:

- 미국 계좌 80,000 USD에서 `Leader2 + Repeat Theme Combo Cap27.5`가 단순 반반 매수가 아닌지 확인한다.
- 한국 주식 10,000,000 KRW에서 초기 3개월 30%, 종목별 상한 22.5%, 정수 주식 주문 현실성을 확인한다.
- ETF 10,000,000 KRW에서 50/40/10 목표 금액과 매수/매도 수량이 맞는지 확인한다.
- 기존 앱 위에 업데이트 설치했을 때 새 APK 내장 데이터가 오래된 원격/캐시에 덮이지 않는지 확인한다.
- 화면 하단 Android 내비게이션, 상태바, 차트, 버튼이 겹치지 않는지 확인한다.

## 팀 역할 문서와의 관계

새 기능을 설계하거나 앱을 업데이트할 때는 다음 순서를 따른다.

1. `android_app_team_roles.md`로 Product, UX, Design, 자산 운용, 데이터, Android, QA 관점을 먼저 통과시킨다.
2. 이 문서의 네 가지 코딩 원칙으로 구현 범위를 제한한다.
3. 대표 금액과 실제 사용 시나리오로 검증한다.
4. 변경 결과를 `android_app_development_log.md`와 필요한 릴리즈 문서에 기록한다.

## 완료 기준

이 규칙을 적용한 작업은 다음 조건을 만족해야 완료로 본다.

- 변경 이유와 범위가 사용자 요청에 직접 연결된다.
- 불필요한 UI, 추상화, 설정값이 추가되지 않는다.
- 전략 계산은 대표 케이스로 검증된다.
- 빌드 또는 실행 가능한 검증 결과가 남는다.
- 문서가 함께 업데이트된다.
