# Native Android App Development Plan

작성일: 2026-07-09

이 문서는 `android_app_strategy_plan.md`의 제품 방향을 기반으로 하되, PWA, TWA, Capacitor 전환 단계를 거치지 않고 바로 네이티브 Android 앱을 개발하는 실행 계획이다. 기존 웹 대시보드는 버리지 않는다. 다만 앱 화면을 그대로 감싸는 대상이 아니라, 전략 로직, 검증 데이터, 화면 요구사항을 추출하는 기준 자료로 사용한다.

## 1. 개발 결정

### 1.1 방향

- 앱 방식: 네이티브 Android
- 권장 기술: Kotlin, Jetpack Compose, Material 3, Room, DataStore, WorkManager
- 제외 범위: PWA, TWA, Capacitor, 웹뷰 중심 앱
- 앱 성격: 자동 주문 앱이 아니라 전략 기반 의사결정 보조 앱
- 첫 MVP 목표: 사용자가 오늘 해야 할 매수, 매도, 리밸런싱, 기록 작업을 Android 앱에서 바로 확인하고 기록할 수 있게 만든다.

### 1.2 왜 네이티브로 바로 가는가

- 푸시 알림, 로컬 스케줄, 보안 저장소, 오프라인 기록, 위젯 같은 Android 기능이 투자 실행 앱의 핵심이다.
- 사용자는 차트 대시보드보다 `오늘 할 일`, `매수 금액`, `매도 예정`, `기록 누락`을 빠르게 확인해야 한다.
- 현재 웹 구현은 제품 검증에는 충분하지만, localStorage 기반 계좌 기록은 장기 운용 앱의 최종 저장 방식으로는 부족하다.
- 네이티브 앱에서 도메인 엔진과 화면 구조를 분리해두면 이후 iOS, 웹, 서버 배치 작업으로도 확장하기 쉽다.

### 1.3 웹 대시보드와 Android 앱의 차이

현재 웹 대시보드는 `전략이 유효한지 검증하고 설명하는 화면`이다. Android 앱은 같은 화면을 옮기는 것이 아니라 `검증된 전략을 오늘 실제 행동으로 바꾸는 실행 도구`가 되어야 한다.

따라서 Android 앱은 아래 원칙으로 다시 설계한다.

- 백테스트 표와 긴 설명보다 오늘 해야 할 매수, 매도, 기록, 경고를 우선한다.
- 종목 추천 이유는 보조 정보이며, 첫 화면의 중심은 `살지/팔지/얼마나/언제/기록했는지`다.
- 웹의 전략 검증 데이터는 앱의 참고 자료와 테스트 fixture로 사용하고, 앱 GUI는 실전 운용 플로우에 맞춰 새로 만든다.
- 앱은 사용자의 실제 계좌, 현금, lot, 체결가, 환율, 수수료를 기준으로 개인화된 주문 가이드를 만든다.
- 앱은 자동 주문을 하지 않지만, 주문 전 체크리스트와 주문 후 기록 누락까지 관리한다.

### 1.4 확정된 제품 결정

이번 기획에서 아래 결정은 확정값으로 둔다.

1. MVP 자산 범위
   - 미국 주식, 한국 주식, 한국 ETF를 모두 포함한다.
   - 세 자산군은 같은 앱 흐름을 쓰되 계좌, 전략 규칙, 주문 가능 시간, 알림 이벤트는 분리한다.
   - 첫 화면에서는 세 자산군의 오늘 할 일을 합쳐 보여주고, 상세 화면에서는 자산군 필터를 제공한다.

2. 자동화 방식
   - 기본 자동화는 `GitHub Actions + GitHub Pages 정적 JSON`으로 한다.
   - 로컬 PC 수동 실행은 백업 운영 방식으로만 둔다.
   - 사용자 개인 계좌 기록은 서버에 올리지 않고 Android Room DB에 저장한다.

3. 알림 정책
   - 사용자가 임의로 모든 알림 시간을 설계하는 방식이 아니라, 전략 이벤트가 알림을 만든다.
   - 월간 신호, 다음 거래일 매수, 6개월 50% 매도, 잔여 50% 주봉 훼손, 12개월 도달, ETF 리밸런싱, 기록 누락이 기본 알림 이벤트다.
   - 사용자는 알림 켜기/끄기와 조용한 시간대만 조정하고, 기본 일정은 전략 규칙에 따른다.

## 5. UX 개발 원칙

### 5.1 첫 화면 원칙

- 첫 화면은 차트가 아니라 `오늘 할 일`이다.
- 사용자가 가장 먼저 보는 문장은 수익률 설명이 아니라 실행 문장이어야 한다.
- 예: `미국 주식 신규 매수 2건`, `6개월 50% 매도 1건`, `ETF 리밸런싱 필요`
- 위험하거나 불완전한 데이터가 있으면 행동 카드보다 위에 경고를 둔다.

### 5.2 투자 시작 원칙

- 투자 시작일 이전 신호는 매수 가이드로 가져오지 않는다.
- 처음 시작한 사용자는 현재 월 확정 후보부터 운용한다.
- 과거 추천 종목은 백테스트와 참고 기록에만 둔다.
- 시작 마법사는 `시장 선택 -> 계좌 설정 -> 전략 선택 -> 현재 월 주문 가이드 -> 체결 기록` 순서로 진행한다.

### 5.3 주문 가이드 원칙

- 앱은 자동 주문을 하지 않는다.
- 주문 전 화면에는 종목, 금액, 예상 수량, 환율, 수수료, 한도 초과 여부, 데이터 최신성을 표시한다.
- 사용자가 증권사 앱에서 주문한 뒤 실제 체결가와 수량을 기록해야 다음 lot이 생긴다.
- 기록하지 않은 주문 가이드는 `기록 누락` 알림으로 남긴다.

### 5.4 보유 lot 원칙

- 같은 종목이 반복 추천되어도 종목 카드는 하나로 합친다.
- 카드 안에는 추천월별 lot을 별도로 쌓는다.
- 6개월 50% 매도와 잔여 50% 주봉 연장 판단은 lot 단위로 추적한다.
- 종목별 한도는 모든 lot의 원금을 합산해 계산한다.

### 5.5 실전 앱 GUI 원칙

웹의 백테스트 대시보드 UI를 그대로 옮기지 않는다. Android 앱의 기본 화면은 아래처럼 재구성한다.

| 앱 화면 | 웹 대시보드와 다른 점 |
|---|---|
| 오늘 | 백테스트 요약 대신 실행 카드만 표시한다. 매수, 매도, 기록 누락, 데이터 지연을 우선순위로 정렬한다. |
| 주문 가이드 | 후보 종목 차트보다 주문 금액, 예상 수량, 한도 초과, 현금 부족, 환율, 수수료를 먼저 보여준다. |
| 보유 lot | 종목 성과표가 아니라 lot별 다음 행동 날짜와 남은 물량 상태를 중심으로 보여준다. |
| 기록 | 사용자가 증권사에서 주문한 뒤 실제 체결가, 수량, 수수료, 환율을 빠르게 입력하도록 만든다. |
| 알림 센터 | 전략 설명이 아니라 아직 처리하지 않은 실행 이벤트를 모아 보여준다. |
| 백테스트 | 실전 흐름을 방해하지 않도록 요약과 신뢰 근거만 제공하고, 기본 탭으로 두지 않는다. |

핵심 GUI 패턴은 `카드 -> 상세 바텀시트 -> 기록 완료` 흐름이다. 예를 들어 `미국 주식 신규 매수 2건` 카드를 누르면 두 종목의 주문 가이드가 나오고, 각 종목에서 `주문 기록`을 누르면 실제 체결 입력 화면으로 이어진다.

### 5.6 디자인/UX 팀 재협의 결과

디자인 팀과 UX 팀의 결론은 `투자 연구 대시보드`가 아니라 `전략 운용 콘솔`로 설계해야 한다는 것이다. 앱의 목적은 종목을 많이 보여주는 것이 아니라, 사용자가 전략대로 운용하고 자산 변화를 놓치지 않게 만드는 것이다.

핵심 사용자 질문:

1. 오늘 내가 반드시 해야 할 일은 무엇인가?
2. 매수 또는 매도해야 한다면 얼마, 몇 주, 언제 해야 하는가?
3. 내 자산은 지난번보다 늘었는가, 줄었는가?
4. 손익 변화가 가격 때문인지, 환율 때문인지, 입출금 때문인지 알 수 있는가?
5. 놓친 매수, 매도, 기록이 있는가?
6. 데이터가 최신이고 믿을 수 있는 상태인가?

UX 팀 결정:

- 첫 화면은 `오늘의 운용 카드`와 `자산 변화 요약`을 같이 보여준다.
- 알림은 앱 밖 푸시만 믿지 않고 앱 안 `Action Inbox`에 영구적으로 남긴다.
- 모든 매수/매도 알림은 `하기 전`, `기록하기`, `완료됨`, `놓침` 상태를 가진다.
- 자산 화면은 총액만 보여주지 않고 `입출금 제외 운용손익`, `환율 영향`, `실현/미실현 손익`, `현금 비중`을 분리한다.
- 매수/매도 화면은 긴 설명보다 실행에 필요한 숫자를 먼저 보여준다.
- 백테스트와 전략 설명은 기본 탭에서 빼고 `전략 신뢰 근거` 화면으로 낮춘다.

디자인 팀 결정:

- 분위기는 화려한 트레이딩 앱보다 차분한 자산 운용 도구에 가깝게 만든다.
- 팔레트는 한 가지 색으로 밀지 않고, 중립 배경 위에 액션/상태 색만 제한적으로 사용한다.
- 카드 안에 카드를 중첩하지 않는다. 화면은 넓은 섹션과 반복 카드로 정리한다.
- 숫자 가독성이 핵심이므로 자산 금액, 수익률, 날짜, 수량은 표 정렬과 고정 폭 숫자 스타일을 우선한다.
- 상승/하락 색은 색만으로 의미를 전달하지 않고 항상 `+`, `-`, `상승`, `하락`, `매수`, `매도` 라벨을 함께 쓴다.
- 한국식 색상 관습과 미국식 색상 관습이 다르므로, `상승=빨강/하락=파랑`과 `상승=초록/하락=빨강`을 설정에서 선택 가능하게 둔다.

### 5.7 추천 정보 구조

기존 하단 내비게이션의 `백테스트` 중심성을 줄이고, 실전 운용에 맞춰 아래 구조를 추천한다.

| 탭 | 목적 | 주요 화면 |
|---|---|---|
| 오늘 | 오늘 놓치면 안 되는 행동 | Action Inbox, 신규 매수, 매도 예정, 기록 누락, 데이터 상태 |
| 자산 | 자산 변화와 계좌 상태 | 총자산, 계좌별 평가액, 현금, 월간 변화, 환율 영향, 손익 분해 |
| 운용 | 보유 종목과 lot 관리 | 종목 카드, lot 리스트, 6개월/12개월 일정, ETF 목표 비중 |
| 기록 | 실제 체결과 입출금 입력 | 매수 기록, 매도 기록, 입금/출금, 환전, 배당, 수수료 |
| 설정 | 전략과 앱 설정 | 전략 선택, 수수료, 환율, 알림, 백업, 색상 관습 |

`전략 신뢰 근거`와 `백테스트 요약`은 설정 또는 운용 화면의 보조 진입점으로 둔다. 매일 들어가는 기본 탭으로 만들지 않는다.

### 5.8 핵심 화면 설계

#### 오늘

구성:

- 상단: 데이터 기준일, 신호 상태, 마지막 동기화 시각
- 1번 카드: 오늘 반드시 처리할 일
- 2번 카드: 이번 달 신규 매수 가이드
- 3번 카드: 매도/리밸런싱 예정
- 4번 카드: 기록 누락
- 하단: 자산 변화 짧은 요약

액션 우선순위:

1. 데이터 지연 또는 검증 필요
2. 이미 지난 매도/리밸런싱
3. 오늘 또는 다음 거래일 매수
4. 매수/매도 기록 누락
5. 7일 이내 예정 알림
6. 참고용 전략/백테스트 업데이트

#### 자산

구성:

- 총자산과 월간 변화
- 계좌별 평가액: 미국 주식, 한국 주식, 한국 ETF/연금
- 현금과 투자 중 자산 분리
- 실현손익, 미실현손익, 입출금 제외 운용손익
- 환율 영향 분리: 달러 기준 수익률과 원화 기준 수익률
- 자산 배분: 계좌별, 전략별, 종목별 상위 비중

자산 변화 표현:

```text
총자산 변화 = 투자 손익 + 환율 효과 + 입금/출금 + 배당/수수료/세금
```

사용자에게는 위 공식을 카드 4개로 쪼개 보여준다. 사용자가 “돈이 왜 늘거나 줄었는지” 바로 이해하는 것이 목적이다.

#### 운용

구성:

- 계좌 필터: 미국 주식, 한국 주식, 한국 ETF
- 종목 카드: 현재 비중, 목표/한도, 수익률, 다음 액션
- lot 상세: 추천월, 매수일, 매수가, 수량, 50% 매도일, 잔여 물량 조건
- ETF 카드: 현재 비중, 목표 비중, 매수/매도 필요 금액

운용 화면은 표보다 카드와 리스트를 기본으로 하되, 숫자가 많은 영역은 정렬된 미니 테이블을 쓴다.

#### 기록

구성:

- 빠른 기록 버튼: 매수, 매도, 입금, 출금, 환전, 배당
- 알림에서 넘어온 경우 해당 종목/lot/금액을 미리 채운다.
- 사용자는 실제 체결가, 수량, 수수료, 환율만 확인해 저장한다.
- 저장 후 앱은 현금, lot, 수익률, 다음 알림을 즉시 재계산한다.

#### 알림 센터

알림 센터는 별도 탭이 아니라 `오늘` 안의 Action Inbox로 둔다. 놓치면 안 되는 작업이므로 앱 안에서도 계속 보이게 한다.

상태:

- 예정
- 오늘
- 지남
- 기록 필요
- 완료
- 보류
- 데이터 검증 필요

알림 종류:

- 월간 신호 확정
- 신규 매수 가이드
- 매수 기록 누락
- 6개월 50% 매도
- 잔여 50% 주봉 훼손 검토
- 12개월 최대 보유기간 도달
- ETF 리밸런싱
- 현금 부족
- 종목 한도 초과
- 환율 급변
- 데이터 지연

### 5.9 디자인 시스템

기본 방향은 Material 3 기반 Compose 디자인 시스템이다. Material 3의 color scheme, typography, shape를 그대로 쓰되 투자 앱에 필요한 상태 토큰을 추가한다.

색상 원칙:

- 배경: 밝은 모드 `#F7F8FA`, 어두운 모드 `#101214`
- 표면: 밝은 모드 `#FFFFFF`, 어두운 모드 `#171A1F`
- 기본 텍스트: 중립 계열 고대비
- 핵심 브랜드 색: 차분한 딥 틸 또는 네이비 계열
- 매수/긍정: 설정 가능한 상승색
- 매도/위험: 설정 가능한 하락색 또는 위험색
- 경고: 앰버 계열
- 정보: 블루 계열
- 데이터 지연: 보라가 아니라 중립+경고 조합으로 표시

테마 원칙:

- 라이트/다크 모드를 모두 지원한다.
- 다크 모드는 검은 배경에 형광색을 쓰는 트레이딩 터미널처럼 만들지 않는다.
- 배경 장식용 그라데이션, 오브, 과한 그림자는 쓰지 않는다.
- 카드 반경은 8dp를 기본으로 한다.
- 액션 버튼은 명확한 동사로 쓴다: `주문 기록`, `매도 기록`, `보류`, `완료`.

타이포그래피:

- 한국어 본문은 Noto Sans KR 또는 시스템 기본 Sans 계열을 사용한다.
- 금액과 수익률은 표 정렬이 가능한 숫자 스타일을 쓴다.
- 큰 숫자는 `headlineSmall` 또는 `titleLarge`, 설명은 `bodyMedium`, 보조 라벨은 `labelMedium`을 기준으로 한다.
- 본문 12sp 미만은 사용하지 않는다.
- 숫자와 라벨을 한 줄에 욱여넣지 않고, 작은 화면에서는 두 줄 구조로 전환한다.

컴포넌트:

- ActionCard: 오늘 해야 할 일
- AssetSummaryCard: 총자산/손익/현금
- OrderGuideSheet: 주문 전 확인 바텀시트
- LotTimeline: lot별 6개월/12개월 진행 상태
- AlertRow: 알림 상태 행
- DataStatusBanner: 데이터 지연/검증 필요
- AccountSegmentedControl: 계좌 전환
- MoneyInputField: 금액 입력
- ExecutionRecordSheet: 체결 기록

### 5.10 알림 UX

알림은 `놓치지 않게`가 목적이지 사용자를 괴롭히는 것이 아니다.

원칙:

- 알림 권한은 앱 첫 실행 직후 묻지 않는다.
- 사용자가 첫 매수 가이드를 만들거나 알림 설정을 켤 때 권한을 요청한다.
- 알림 권한이 꺼져 있어도 앱 안 Action Inbox는 계속 동작한다.
- 중요한 알림은 로컬 알림과 앱 안 배지를 함께 사용한다.
- 완료된 알림은 사라지는 것이 아니라 기록으로 남긴다.

권장 알림 채널:

| 채널 | 긴급도 | 예시 |
|---|---|---|
| 실행 필요 | 높음 | 오늘 매수, 오늘 매도, 리밸런싱 |
| 기록 필요 | 보통 | 주문 가이드 생성 후 체결 기록 누락 |
| 데이터 상태 | 보통 | 데이터 지연, 검증 필요 |
| 참고 | 낮음 | 백테스트 요약 업데이트, 전략 설명 변경 |

알림 문구 예시:

- `미국 주식 신규 매수 2건: TECH, KLAC`
- `STX 2026-01 lot 50% 매도일입니다`
- `한국 ETF 리밸런싱 필요: 반도체 비중 +4.2%`
- `매수 기록이 아직 없습니다: AMD 주문 여부를 기록하세요`

### 5.12 전략 기반 알림 정책

알림은 전략 규칙에서 자동 생성한다. 사용자는 알림 시간을 세세하게 설계하지 않고, 앱은 각 이벤트의 성격과 시장 시간에 맞춰 기본 알림을 만든다.

공통 원칙:

- 알림은 `예정 -> 오늘 -> 기록 필요 -> 완료` 상태를 가진다.
- 앱 푸시를 놓쳐도 Action Inbox에 남는다.
- 데이터가 지연되면 매수/매도 알림을 확정하지 않고 `데이터 검증 필요` 알림을 먼저 만든다.
- 사용자가 이미 매수/매도 기록을 완료한 lot에는 같은 실행 알림을 반복하지 않는다.
- 시장 개장 전 알림은 실제 주문 전 체크리스트로 연결하고, 장마감 후 알림은 기록 누락 확인으로 연결한다.

전략별 기본 알림:

| 전략/자산군 | 이벤트 | 기본 알림 시점 | 사용자 행동 |
|---|---|---|---|
| 미국 주식 | 월간 후보 확정 | 월말 데이터 검증 완료 후 | 이번 달 후보 확인 |
| 미국 주식 | 신규 매수 가이드 | 다음 미국 거래일 개장 전 | 주문 금액/예상 수량 확인 |
| 미국 주식 | 매수 기록 누락 | 매수 예정일 장마감 후에도 기록이 없을 때 | 실제 체결 여부 기록 |
| 미국 주식 | 6개월 50% 매도 | 7일 전, 1일 전, 당일 개장 전 | lot 50% 매도 검토 및 기록 |
| 미국 주식 | 잔여 50% 주봉 훼손 | 주봉 데이터 검증 완료 후 | 최종 매도 검토 |
| 미국 주식 | 12개월 도달 | 7일 전, 1일 전, 당일 개장 전 | 잔여 lot 정리 검토 |
| 한국 주식 | 월간 후보 확정 | 한국 월말 데이터 검증 완료 후 | 이번 달 후보 확인 |
| 한국 주식 | 신규 매수 가이드 | 다음 한국 거래일 개장 전 | 주문 금액/예상 수량 확인 |
| 한국 주식 | 6개월 50% 매도 | 7일 전, 1일 전, 당일 개장 전 | lot 50% 매도 검토 및 기록 |
| 한국 주식 | 잔여 50% 주봉 훼손 | 주봉 데이터 검증 완료 후 | 최종 매도 검토 |
| 한국 ETF | 월간 목표 비중 확정 | 한국 월말 데이터 검증 완료 후 | 목표 비중 확인 |
| 한국 ETF | 리밸런싱 필요 | 다음 한국 거래일 개장 전 | 매수/매도 금액 확인 |
| 한국 ETF | 리밸런싱 기록 누락 | 리밸런싱 예정일 장마감 후에도 기록이 없을 때 | 실제 체결 여부 기록 |

알림 강도:

- 높음: 당일 매수/매도/리밸런싱, 이미 지난 실행 이벤트
- 보통: 7일 전 매도 예정, 기록 누락, 데이터 검증 필요
- 낮음: 월간 후보 확정, 백테스트 요약 업데이트

사용자 설정:

- 조용한 시간대
- 알림 사전 예고: 7일 전 사용 여부
- 상승/하락 색상 관습
- 민감 정보 숨김: 알림에 금액과 종목명을 숨기는 모드

### 5.13 위젯과 잠금화면 확장

MVP 이후에는 Android 홈 화면 위젯을 제공한다.

위젯 후보:

- 오늘 할 일 개수
- 다음 매수/매도 예정일
- 데이터 최신성 상태
- 총자산 전일/전월 변화

위젯에서는 민감한 금액을 숨기는 모드를 기본 제공한다. 예: `오늘 할 일 3건`, `매도 예정 1건`처럼 금액 없이 표시한다.

### 6.3 로컬 우선 저장

MVP는 로그인 없이도 사용할 수 있는 로컬 우선 앱으로 시작한다.

- 계좌, lot, 거래, 입출금, 수수료 프로필은 Room에 저장한다.
- 알림 설정과 앱 설정은 DataStore에 저장한다.
- 민감한 설정은 Android Keystore 기반 암호화 저장소를 사용한다.
- 백업/복원은 JSON 내보내기와 가져오기로 제공한다.
- 서버 동기화는 v1.1 이후 붙이되, 데이터 모델은 처음부터 동기화 가능한 ID와 updatedAt을 포함한다.

### 8.2 포팅 검증 방식

- 현재 JS 산출물의 JSON을 Android 테스트 fixture로 고정한다.
- Kotlin 엔진이 같은 입력에서 같은 매수 후보, 주문 금액, lot 상태, 매도 예정일을 내는지 비교한다.
- 미국 주식, 한국 주식, 한국 ETF를 각각 최소 20개 케이스로 검증한다.

### 8.3 매월 종목 추천 생성 방식

앱이 사용자 휴대폰 안에서 수많은 종목 데이터를 직접 긁어와 매월 전체 스크리닝을 돌리는 구조는 MVP의 기본 방식으로 두지 않는다. 기술적으로는 가능하지만, 데이터 라이선스, 통신량, 배터리, 실패 복구, 시장 휴장일 처리, 가격 데이터 검증 때문에 실전 앱에는 불안정하다.

실전 구조는 `서버 신호 생성 + 앱 개인화 계산`이다.

```text
서버/데이터 파이프라인
  1. 유니버스 로드: 미국 주식, 한국 주식, 한국 ETF
  2. 가격/거래대금/환율 데이터 수집
  3. 누락, 이상치, 휴장일, 상장폐지, 티커 변경 검증
  4. 월말 기준 주도 섹터/업종 랭킹 계산
  5. active 전략 규칙으로 월간 후보 확정
  6. Signal을 recommendationMonth, asOf, nextTradingDay와 함께 고정
  7. 앱 API 또는 정적 signal package로 배포

Android 앱
  1. 최신 확정 Signal 동기화
  2. 사용자 계좌, 현금, 보유 lot, 환율, 수수료, 종목 한도 반영
  3. 매수 가능 금액과 예상 수량 계산
  4. 주문 전 체크리스트 표시
  5. 사용자가 실제 체결을 기록하면 lot 생성
```

사용자 입장에서는 앱이 자동으로 이번 달 추천을 보여주지만, 실제 대량 데이터 검토와 후보 확정은 서버 작업이 담당한다. 앱은 그 신호를 사용자의 실제 계좌 상태에 맞게 실행 가능한 금액과 액션으로 바꾼다.

월간 추천은 아래 원칙을 따른다.

- 월말 기준으로만 확정한다.
- 확정 후에는 월중 순위 변화로 후보를 자동 교체하지 않는다.
- 가격 데이터가 불완전하면 새 신호를 만들지 않고 이전 신호 유지와 `데이터 지연` 경고를 표시한다.
- 미국 주식은 `Leader2 + Repeat Theme Combo Cap27.5`를 active 기본 전략으로 사용한다.
- 한국 주식과 한국 ETF도 같은 신호 배포 구조를 쓰되, 전략 규칙은 별도 객체로 분리한다.
- 앱은 사용자의 투자 시작월 이전 신호를 매수 대상으로 표시하지 않는다.

### 8.4 앱 자체 추천 엔진의 범위

앱 안에도 전략 엔진은 필요하다. 다만 역할은 `전체 시장 스크리닝`이 아니라 `확정 신호의 실전 적용`이다.

앱 내부에서 계산할 것:

- 현재 계좌 기준 매수 가능 금액
- 종목별 누적 원금 한도 초과 여부
- 반복 추천 종목의 lot 병합 표시
- 소수점 거래 가능 여부에 따른 예상 수량
- 현금 부족, 최소 주문 미달, 환율 급변 경고
- 입금/출금 후 다음 신호 반영 방식
- 실제 체결가 기준 수익률과 다음 매도 일정

서버 또는 데이터 파이프라인이 계산할 것:

- 전체 유니버스 가격 수집
- 주도 섹터/업종 랭킹
- 월간 후보 확정
- ETF 목표 비중 확정
- 대량 가격 데이터 기반 주봉 지표 계산
- 데이터 품질 검증과 신호 발행 여부 결정

장기적으로는 앱에 `오프라인 검산 모드`를 둘 수 있다. 이 모드는 서버가 내려준 원본 데이터 일부로 계산 결과를 검산하는 용도이며, 공식 신호 생성의 주체는 서버로 둔다.

### 8.5 주봉 훼손 포착 구현

주봉 훼손 포착은 구현 가능하다. 다만 실전 앱에서는 장중 흔들림이 아니라 `주봉 마감 기준`으로 판단해야 불필요한 매도 알림을 줄일 수 있다.

기본 규칙:

- lot 매수 후 6개월 전까지는 주봉 훼손만으로 매도하지 않는다.
- 6개월 도달 시 해당 lot의 50% 매도 알림을 만든다.
- 잔여 50%는 6개월 시점의 주봉 추세가 살아 있을 때만 연장 보유한다.
- 연장 보유 중에는 주봉 10주선 2주 연속 이탈 또는 주봉 RSI 50 하회를 감지하면 최종 매도 검토 알림을 만든다.
- 최대 12개월 도달 시 잔여 물량 정리 알림을 만든다.

구현 흐름:

```text
weeklyTrendCheck(lot, weeklyBars):
  if lot.age < 6 months:
    return HOLD_NO_WEEKLY_EXIT

  if lot.halfSold == false:
    return ALERT_HALF_SELL

  latestClosedWeek = weeklyBars.latestClosed()
  previousClosedWeek = weeklyBars.previousClosed()

  below10wTwoWeeks =
    latestClosedWeek.close < latestClosedWeek.ma10
    and previousClosedWeek.close < previousClosedWeek.ma10

  rsiBroken = latestClosedWeek.rsi14 < 50
  maxReached = today >= lot.maxExitDate

  if below10wTwoWeeks or rsiBroken or maxReached:
    return ALERT_FINAL_SELL_REVIEW

  return KEEP_EXTENDED
```

데이터 처리 원칙:

- 주봉은 서버가 일봉을 집계해 계산하거나, 검증된 주봉 OHLC를 내려준다.
- 앱은 최신 주봉 마감 데이터와 lot 상태를 합쳐 알림을 표시한다.
- 금요일 장마감 직후 데이터가 불완전할 수 있으므로, 다음 거래일 데이터 검증 후 알림을 보내는 방식을 기본값으로 둔다.
- 가격 데이터가 없으면 매도 알림을 확정하지 않고 `주봉 데이터 검증 필요` 상태로 둔다.
- 사용자가 수동으로 매도했다고 기록하면 해당 lot의 알림을 종료한다.

### 8.6 월간 추천과 주봉 감지의 MVP 판정

MVP에서 반드시 해야 하는 것:

- 최신 확정 월간 신호를 받아 앱에 표시
- 계좌 상태에 맞춰 매수 금액과 수량 계산
- 사용자가 체결 기록을 남기면 lot 생성
- lot별 6개월 50% 매도 알림 생성
- 주봉 훼손 또는 12개월 도달 시 잔여 매도 검토 알림 생성
- 데이터 누락 시 새 추천과 매도 판단을 중단하고 경고 표시

MVP에서 하지 않는 것:

- 휴대폰이 수천 개 종목을 직접 스크래핑해서 공식 월간 후보를 확정하는 기능
- 장중 실시간 틱 데이터로 매수/매도 신호를 바꾸는 기능
- 증권사 자동 주문 기능
- 백테스트 전략 탐색을 앱 안에서 새로 수행하는 기능

## 12. 백엔드 전환 계획

네이티브 앱으로 바로 가더라도 서버는 필요하다. 다만 MVP에서는 서버를 필수로 만들지 않고, 현재 생성되는 JSON 데이터를 앱에서 읽을 수 있는 형태로 정리한다.

### 12.0 무료 우선 서버 전략

최대한 무료로 운영하려면 처음부터 사용자별 서버 DB와 상시 API 서버를 만들지 않는다. MVP는 `정적 신호 패키지 + 앱 로컬 DB` 구조로 시작한다.

핵심 원칙:

- 사용자 계좌, 체결 기록, 현금, lot, 수수료, 환율 설정은 Android 앱의 Room DB에만 저장한다.
- 서버에는 개인 투자 기록을 저장하지 않는다.
- 월간 종목 추천과 주봉 지표는 모든 사용자에게 같은 `공식 신호 패키지`로 배포한다.
- 앱은 신호 패키지를 다운로드한 뒤, 사용자 계좌 상태에 맞춰 매수 금액과 알림을 로컬에서 계산한다.
- 푸시 서버는 MVP에서 만들지 않고, WorkManager와 로컬 알림으로 처리한다.
- 로그인과 다중 기기 동기화는 비용이 생길 수 있으므로 v1 이후 선택 기능으로 미룬다.

무료 MVP 구성:

```text
GitHub Actions
  -> 월말/주간 데이터 스크립트 실행
  -> signal package JSON 생성

GitHub Pages 또는 정적 호스팅
  -> /api/manifest.json
  -> /api/signals/latest.json
  -> /api/signals/2026-07.json
  -> /api/weekly-trends/latest.json
  -> /api/backtests/summary.json

Android 앱
  -> 정적 JSON 다운로드
  -> Room DB의 사용자 계좌와 결합
  -> 오늘 할 일, 주문 가이드, lot 알림 계산
```

로컬 PC 실행은 GitHub Actions 실패 시 수동 복구용으로만 둔다.

이 구조에서는 앱 입장에서 서버 API처럼 보이지만, 실제로는 정적 JSON 파일을 받는 것뿐이다. 그래서 서버 인스턴스, 데이터베이스, 인증 서버가 없어도 시작할 수 있다.

### 12.0.1 무료 구성별 역할

| 구성 | 무료 우선 역할 | 비용 발생을 늦추는 이유 |
|---|---|---|
| Android Room | 사용자 계좌와 거래 기록 저장 | 사용자별 클라우드 DB가 필요 없다 |
| WorkManager | 6개월 매도, 주봉 점검, 기록 누락 로컬 알림 | 푸시 서버 없이 알림 가능 |
| GitHub Actions | 월말 신호와 주간 추세 JSON 생성 | 스케줄 배치 서버를 따로 운영하지 않는다 |
| GitHub Pages | 앱이 받을 정적 JSON 호스팅 | 상시 API 서버가 필요 없다 |
| Firebase FCM | v1 이후 선택적 푸시 | MVP에서는 쓰지 않고 필요할 때만 추가 |
| Cloudflare Workers/D1 | v1 이후 가벼운 API/DB 대안 | 정적 JSON으로 부족해질 때만 검토 |

### 12.0.2 무료 MVP에서 만들 API 파일

```text
/api/manifest.json
  최신 신호 월, 데이터 생성 시각, 데이터 품질 상태

/api/signals/latest.json
  미국 주식, 한국 주식, 한국 ETF의 최신 확정 신호

/api/signals/YYYY-MM.json
  특정 월 확정 후보와 nextTradingDay

/api/weekly-trends/latest.json
  보유 점검에 필요한 주봉 10주선, RSI, 훼손 여부

/api/fx/latest.json
  USD/KRW 기준 환율, 데이터 기준 시각

/api/backtests/summary.json
  앱에서 보여줄 최소 백테스트 요약
```

앱은 `manifest.json`만 먼저 확인하고, 새 버전이 있을 때 필요한 JSON만 내려받는다. 데이터가 없거나 오래되면 새 매수/매도 판단을 확정하지 않고 `데이터 지연` 상태를 보여준다.

### 12.0.3 무료 운영 단계

1. 완전 무료 수동 운영
   - 개발자가 로컬 PC에서 기존 Node 스크립트를 실행한다.
   - 생성된 JSON을 GitHub Pages에 배포한다.
   - 백업 운영 방식이다. 가장 단순하고 비용이 없지만, 월말/주간 업데이트를 사람이 챙겨야 한다.

2. 거의 무료 자동 운영
   - GitHub Actions 스케줄로 월말/주간 스크립트를 실행한다.
   - 결과 JSON을 Pages에 자동 배포한다.
   - 개인 사용 기본 방식으로 채택한다.

3. 무료 티어 서버리스 운영
   - 정적 JSON으로 부족해질 때 Cloudflare Workers, Firebase, Supabase 같은 무료 티어를 검토한다.
   - 로그인, 클라우드 백업, 사용자별 동기화가 필요해질 때만 적용한다.

### 12.0.4 개인용 권장 운영안

개인적으로만 사용할 앱이라면 서버는 `자동 계산기` 역할만 하면 된다. 사용자 계좌 기록을 서버에 저장할 필요가 없고, 앱 배포도 당장 고려하지 않으므로 가장 단순한 운영안은 아래와 같다.

권장안:

- GitHub Actions가 정해진 시간에 기존 Node 전략 스크립트를 실행한다.
- 스크립트가 미국 주식, 한국 주식, 한국 ETF 데이터를 수집하고 active 전략으로 후보를 고른다.
- 결과를 `docs/api` 또는 `dist/api` 같은 정적 폴더에 JSON으로 저장한다.
- GitHub Pages가 해당 JSON을 호스팅한다.
- Android 앱은 앱을 열 때 또는 하루 1회 `manifest.json`을 확인하고 새 신호가 있으면 내려받는다.
- 사용자 계좌 기록과 주문 가이드 계산은 앱 안에서만 처리한다.

자동 종목 선정은 가능하다. 다만 `항상 켜진 서버가 실시간으로 감시`하는 것이 아니라 `스케줄러가 정해진 시각에 배치 작업을 실행`하는 방식이다.

```text
월말 장마감 이후
  -> 가격 데이터 수집
  -> 데이터 누락/이상치 검증
  -> 주도 섹터/업종 계산
  -> 월간 후보 확정
  -> /api/signals/YYYY-MM.json 갱신

매주 주봉 마감 이후
  -> 보유 후보군 weekly bar 갱신
  -> 10주선, RSI, 2주 연속 이탈 여부 계산
  -> /api/weekly-trends/latest.json 갱신

앱 실행 시
  -> manifest 확인
  -> 새 JSON 다운로드
  -> 내 Room DB의 lot과 결합
  -> 오늘 할 일과 알림 재계산
```

개인용 기본 스케줄 예시:

| 작업 | 권장 실행 시점 | 산출물 |
|---|---|---|
| 미국 주식 월간 신호 | 미국 월말 장마감 후 또는 다음 날 오전 | `/api/signals/us/latest.json` |
| 한국 주식 월간 신호 | 한국 월말 장마감 후 또는 다음 날 저녁 | `/api/signals/kr-stock/latest.json` |
| 한국 ETF 리밸런싱 신호 | 한국 월말 장마감 후 또는 다음 거래일 전 | `/api/signals/kr-etf/latest.json` |
| 주봉 훼손 점검 | 매주 주봉 마감 후 다음 날 | `/api/weekly-trends/latest.json` |
| 환율 갱신 | 하루 1회 또는 앱 수동 갱신 | `/api/fx/latest.json` |

주의할 점:

- GitHub Actions의 cron은 UTC 기준으로 잡아야 하므로 한국/미국 장마감 시각을 UTC로 변환해야 한다.
- 무료 데이터 소스는 실패할 수 있으므로 실패 시 기존 신호를 덮어쓰지 않는다.
- 데이터가 불완전하면 `dataStatus: delayed` 또는 `needs_review`를 manifest에 기록한다.
- 앱은 `dataStatus`가 정상일 때만 새 추천을 확정 신호로 보여준다.
- 개인용이라도 자동 선정 결과를 바로 맹신하지 않고, 앱에 `데이터 기준일`과 `검증 상태`를 표시한다.

### 12.0.5 무료 운영에서 포기할 것

- 사용자별 클라우드 동기화
- 서버 푸시 알림
- 앱 사용자별 서버 계산
- 실시간 가격 스트리밍
- 대량 사용자를 위한 고가용성 API
- 증권사 주문 연동

이 기능들은 무료 MVP의 목표와 맞지 않는다. 대신 앱은 로컬 기록과 정적 신호 패키지로 실전 투자 실행에 집중한다.

### 12.0.6 비용이 생길 가능성이 높은 지점

- 가격 데이터 소스가 무료 제공 범위를 넘을 때
- 앱 사용자가 많아져 정적 파일 다운로드량이 커질 때
- 로그인/클라우드 백업을 붙일 때
- 서버 푸시 알림을 사용자별로 정교하게 보내려 할 때
- Play Store 등록, 앱 서명/배포, 개인정보 처리방침 사이트 운영이 필요할 때
- 데이터 제공처의 약관상 상업적 재배포가 제한될 때

따라서 1차 앱은 `개인 사용/소규모 베타`를 전제로 설계하고, 데이터 소스와 배포 범위가 커질 때 비용 구조를 다시 검토한다.

### MVP

- 서버가 발행한 최신 Signal package 또는 앱 내부 샘플 Signal 사용
- 사용자의 실제 계좌 기록은 로컬 DB 저장
- 가격/환율이 없으면 `데이터 지연` 상태 표시
- 월간 후보 확정은 앱 로컬 계산이 아니라 외부 데이터 파이프라인 산출물로 취급

### v1.1

- Node 기반 기존 데이터 생성 스크립트를 서버 배치로 전환
- 월간 신호 생성과 주간 보유 점검을 서버에서 실행
- 앱은 결과만 내려받고 로컬 계좌 기록과 합쳐 오늘 할 일을 계산

### v1.2

- 로그인, 클라우드 백업, 다중 기기 동기화
- 서버 푸시 알림
- 가격/환율 자동 갱신

| 한국/미국 계좌 규칙 혼합 | Account.market과 Strategy.market을 분리하고 화면도 자산군별 필터링한다 |

## 14. Definition of Done

기능 하나가 완료되려면 아래 조건을 만족해야 한다.

- UX 플로우와 디자인 시안이 연결되어 있다.
- Compose 화면이 작은 화면과 큰 글자 모드에서 깨지지 않는다.
- 도메인 계산은 unit test가 있다.
- 주요 상태는 로딩, 빈 상태, 오류, 데이터 지연 상태를 가진다.
- 투자 조언 오해를 줄이는 문구가 필요한 위치에 있다.
- 실제 계좌 기록과 백테스트 데이터가 섞이지 않는다.
- 알림이 필요한 기능은 알림 권한 거부 상태까지 처리한다.

## 16. 최종 결론

바로 Android 앱을 만든다면 목표는 웹 대시보드 복제가 아니라 `전략 실행 앱`이다. 기존 웹은 검증 자료와 도메인 참고 구현으로 사용하고, 앱은 네이티브의 장점인 로컬 DB, 알림, 오프라인 기록, 보안 저장소, 빠른 실행 흐름을 중심으로 설계한다.

첫 릴리즈는 로그인 없는 로컬 우선 Android 앱으로 충분하다. 사용자가 계좌를 만들고, 현재 월 공식 전략 신호를 확인하고, 주문 후 실제 체결을 기록하고, 6개월/주봉/ETF 리밸런싱 알림을 받을 수 있으면 MVP는 성공이다. 이후 서버 동기화와 푸시 알림을 붙여 장기 운용 앱으로 확장한다.
